# Notebook管线规范

> 将任意场景产出转化为可交互的教学/演示 Jupyter Notebook。

## 定位

Notebook管线是场景E的第6条管线，核心价值是**将项目代码转化为可运行的教学过程**。

与其他管线的关系：
- 项目代码是**源**，notebook是**视图**
- notebook不是开发模式，是开发成果的表达形式
- 所有场景（A/A2/B/C/D）的产出都可以走notebook管线

## 输入来源

| 来源 | 提取内容 | notebook侧重 |
|------|---------|-------------|
| A 知识探索 | 概念推导笔记 | 概念→公式→验证 |
| A2 知识拆解 | 拆解文档+HTML | 知识点→练习→检验 |
| B 蒸馏 | 蒸馏Skill | 思维方法→案例→应用 |
| C 项目开发 | 设计文档+代码 | 概念→实现→测试 |
| D 项目解析 | 解析文档+代码 | 模块→原理→验证 |

## Cell类型规范

### Markdown Cell

用途：解释"为什么"，不是"做了什么"。

结构：
```markdown
## {步骤序号}. {功能名称}

{一句话说明这个步骤解决什么问题}

**设计决策**：{为什么这样做，而不是其他方式}
```

### Code Cell

用途：可运行的代码片段。

规则：
- 每个 cell 只做一件事
- 必须可独立运行（或仅依赖前序 cell）
- 关键步骤后有 print/assert
- 删除非教学代码（日志、边界处理、异常捕获等）
- import 集中在最前面的 cell

### Output Cell

用途：代码运行结果，由实际执行填充。

规则：
- 不手写输出，必须实际运行生成
- 输出必须与代码一致
- 失败输出保留（教学价值）

## 教学结构模板

### 项目开发类（场景C/D）

```
# {项目名称}

## 0. 环境准备
[Code] import + 配置

## 1. {核心概念}
[Markdown] 概念解释
[Code] 最小实现
[Code] 验证

## 2. {第二个概念}
[Markdown] 为什么需要
[Code] 扩展实现
[Code] 验证

## N. 完整系统
[Code] 组合所有模块
[Code] 端到端测试

## N+1. 总结
[Markdown] 关键收获 + 扩展方向
```

### 知识探索类（场景A/A2）

```
# {知识主题}

## 1. 直觉理解
[Markdown] 类比解释
[Code] 简单演示

## 2. 形式化定义
[Markdown] 数学定义
[Code] 从零实现

## 3. 验证与实验
[Code] 参数变化实验
[Code] 边界条件测试

## 4. 总结
[Markdown] 核心要点
```

### 蒸馏类（场景B）

```
# {人物}的{方法}

## 1. 思维模型
[Markdown] 核心心智模型

## 2. 决策启发式
[Markdown] 关键决策规则
[Code] 案例演示

## 3. 场景应用
[Code] 不同场景的路由
[Code] 验证

## 4. 总结
[Markdown] 可迁移的要点
```

## 质量检查

| 检查项 | 标准 |
|-------|------|
| 可运行 | 所有 Code cell 从头到尾顺序运行无报错 |
| 渐进式 | 每个 cell 建立在前一个之上 |
| 有验证 | 关键步骤后有 assert 或 print |
| 有解释 | Markdown cell 解释"为什么" |
| 可修改 | 读者修改参数后能观察不同结果 |
| 无冗余 | 删除了非教学代码 |
| import集中 | 所有 import 在最前面的 cell |

## 输出

- 文件格式：`.ipynb`
- 命名规则：`{序号}-{项目名}.ipynb`（如 `01-build-agent.ipynb`）
- 存放位置：项目根目录或 `notebooks/` 目录
