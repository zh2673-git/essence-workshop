# Skill 质量 Rubric（9+1维度，总分110）

> **设计依据**：基于 Darwin Skill 9维Rubric（源自 Microsoft Research SkillLens 论文 arXiv 2605.23899）+ SkillOpt 验证机制（arXiv 2605.23904），适配本质工坊语境增加第10维度。
>
> **核心发现**：LLM-as-judge 准确率仅 46.4%，加入 meta-skill 三维度后提升到 73.8%。重要决策必须人审。

## 评分维度

### 结构维度（65分）— 静态分析

| # | 维度 | 权重 | 评分标准 |
|---|------|------|---------|
| 1 | **Frontmatter质量** | 7 | name规范、description包含做什么+何时用+触发词、≤1024字符、**禁止结尾加"灵活应用/根据情况判断"等空话尾巴** |
| 2 | **工作流清晰度** | 12 | 步骤明确可执行、有序号、每步有明确输入/输出 |
| 3 | **失败模式编码** | 12 | **必须显式编码失败模式**（写出"如果X失败→Y"的明确分支）；有fallback路径、错误恢复；**只写正向流程而不写失败分支扣≥3分** |
| 4 | **检查点设计** | 6 | 关键节点有可见性输出（🔵 AUTO-CHECK自动通行+摘要 / 🔴 CHECKPOINT关键决策确认），防止自主失控；**检查点必须显性标记（🔴/🔵），仅靠"如果...建议..."措辞不算** |
| 5 | **可执行具体性** | 17 | 不模糊、有具体参数/格式/示例、可直接执行；**禁止"建议/可以考虑/根据情况/灵活把握/视情况而定"等软化措辞**——出现≥3处扣≥3分 |
| 6 | **资源整合度** | 4 | references/scripts/assets引用正确、路径可达 |
| 7 | **整体架构** | 7 | 结构层次清晰、不冗余不遗漏、与本质工坊三层架构一致；**冗余/AI腔废话段落出现一处扣1分** |

### 效果维度（35分）— 需要实测

| # | 维度 | 权重 | 评分标准 |
|---|------|------|---------|
| 8 | **实测表现** | 23 | 用测试prompt跑一遍，输出质量是否符合skill宣称的能力 |

## 评分规则

- 维度1-7、9-10：每个维度打 1-10 分，乘以权重得到该维度得分
- 维度8（实测表现）：跑2-3个测试prompt，按输出质量打1-10分
- **总分 = Σ(维度分 × 权重) / 10**，满分110
- 改进后总分必须**严格高于**改进前才保留

### 分数等级

| 等级 | 分数范围 | 含义 |
|------|---------|------|
| S | ≥95 | 优秀，可直接使用 |
| A | 80-94 | 良好，小修即可 |
| B | 65-79 | 及格，有明显短板需补强 |
| C | 50-64 | 不及格，需大幅改进 |
| D | <50 | 严重不足，建议重写 |

## 各维度详细评分指引

### dim1: Frontmatter质量（7分）

| 分数 | 标准 |
|------|------|
| 9-10 | name简洁规范，description完整包含做什么+何时用+中英文触发词，≤1024字符，无空话尾巴 |
| 7-8 | description基本完整，触发词有但不够全，无空话尾巴 |
| 5-6 | description缺少触发词或用途说明，或有空话尾巴 |
| 1-4 | name不规范，description严重缺失或全是空话 |

### dim2: 工作流清晰度（12分）

| 分数 | 标准 |
|------|------|
| 9-10 | 每步有序号，输入/输出明确，步骤间逻辑清晰，可直接按步执行 |
| 7-8 | 步骤有序号，大部分输入/输出明确，个别步骤模糊 |
| 5-6 | 步骤有但缺序号，或多个步骤输入/输出不明确 |
| 1-4 | 步骤混乱，无法按序执行 |

### dim3: 失败模式编码（12分）

| 分数 | 标准 |
|------|------|
| 9-10 | 每个关键步骤都有"如果X失败→Y"的显式分支，fallback路径完整 |
| 7-8 | 大部分步骤有失败分支，个别遗漏 |
| 5-6 | 只有1-2个步骤有失败分支，其余只写正向流程 |
| 1-4 | 完全没有失败模式编码，只有happy path |

### dim4: 检查点设计（6分）

| 分数 | 标准 |
|------|------|
| 9-10 | 关键节点有🔵 AUTO-CHECK（自动通行+摘要输出）和🔴 CHECKPOINT（关键决策确认），可见性强 |
| 7-8 | 有检查点但部分用文字描述而非显性标记 |
| 5-6 | 检查点数量不足，或仅用"建议确认"措辞 |
| 1-4 | 无检查点，或检查点形同虚设 |

### dim5: 可执行具体性（17分）

| 分数 | 标准 |
|------|------|
| 9-10 | 每步有具体参数/格式/示例，可直接执行，零软化措辞 |
| 7-8 | 大部分步骤具体，1-2处软化措辞 |
| 5-6 | 多个步骤模糊，软化措辞≥3处 |
| 1-4 | 大量模糊描述，无法直接执行 |

### dim6: 资源整合度（4分）

| 分数 | 标准 |
|------|------|
| 9-10 | 所有references/scripts/assets路径可达，引用正确 |
| 7-8 | 个别路径不可达或引用缺失 |
| 5-6 | 多处路径错误或引用缺失 |
| 1-4 | 资源引用严重缺失或全部不可达 |

### dim7: 整体架构（7分）

| 分数 | 标准 |
|------|------|
| 9-10 | 结构层次清晰，与本质工坊三层架构一致，零冗余零AI腔 |
| 7-8 | 结构基本清晰，1处冗余或AI腔 |
| 5-6 | 结构有冗余，多处AI腔表达 |
| 1-4 | 结构混乱，大量冗余和AI腔 |

### dim8: 实测表现（23分）

评分方式：
1. 为skill设计2-3个**典型用户prompt**（最常见使用场景）
2. 带skill执行 vs 不带skill执行（baseline）
3. 对比输出质量打分

| 分数 | 标准 |
|------|------|
| 9-10 | 带skill输出明显优于baseline，完成用户意图，无负面影响 |
| 7-8 | 带skill输出优于baseline，但有轻微过度冗余 |
| 5-6 | 带skill输出与baseline持平，skill价值不明显 |
| 1-4 | 带skill输出不如baseline，skill引入负面影响 |

**若子agent不可用**：退化为干跑验证（读完skill后模拟执行思路），必须标注 `dry_run`。dry_run比例>30%→评估失效警告。

### dim9: 反例与黑名单（6分）

| 分数 | 标准 |
|------|------|
| 9-10 | 有完整的"不要做什么"清单，≥5条具体反模式，每条有替代做法 |
| 7-8 | 有反例清单但条目不足5条，或部分不够具体 |
| 5-6 | 只有1-2条反例，或反例太笼统 |
| 1-4 | 完全没有反例清单 |

### dim10: 本质工坊一致性（6分）

| 分数 | 标准 |
|------|------|
| 9-10 | 坡度理解贯穿（点出→联系→详细），三阶方法论完整应用，类-属性-方法-路由模型使用 |
| 7-8 | 坡度理解基本贯穿，三阶方法论应用但个别阶段缺失 |
| 5-6 | 坡度理解不完整，三阶方法论只应用了1-2阶 |
| 1-4 | 未使用坡度理解和三阶方法论 |

## High-Leverage 操作速查

来自 Darwin Skill 实战验证的4条高效操作：

- **HL-1（dim4）显性视觉标记是杠杆**：加� AUTO-CHECK / �🔴 CHECKPOINT，靠"必须"措辞不行——LLM解析时扫描视觉标记
- **HL-2（dim3）if-then三段式fallback表**：把"症状/解法"升级为"触发条件/一线修复/仍失败兜底"三段式
- **HL-3（诊断）维度相关簇警告**：dim2/3/4是相关簇——修dim3时dim2常跟着涨
- **HL-4（退出）触顶自动break**：连续2轮Δ<2分→停止优化，见好就收
