# 知本（KnowledgeRoot）- 项目式拆解与知识图谱感知设计

## 一、现状分析

### 当前能力

| 能力 | 状态 | 说明 |
|------|------|------|
| 单知识点七维拆解 | 已实现 | 输入"负数"，输出7个维度的HTML |
| SVG图示生成 | 已实现 | 按范式族匹配图示类型 |
| 知识图谱JSON存储 | 已实现 | nodes + edges 结构 |
| 知识地图链接 | 部分实现 | 已生成节点可点击，未生成节点pending |
| 递归生成 | 已定义流程 | 手动/自动1层/自动N层 |

### 当前缺陷

| 缺陷 | 影响 | 严重程度 |
|------|------|---------|
| **无法拆解复杂问题/项目** | 只能处理单个知识点，无法实现项目式学习 | 高 |
| **知识图谱感知缺失** | 生成新知识点时不扫描已有知识，可能重复生成 | 高 |
| **已有HTML不自动更新** | 生成新知识点后，旧HTML中的pending节点不会变成可点击链接 | 中 |
| **无全局知识图谱视图** | 无法一览所有已生成知识的全貌 | 中 |

## 二、项目式拆解设计

### 核心思想

**知识点拆解**回答"这个概念是什么"，**项目式拆解**回答"这个问题怎么解决"。

项目式拆解不是替代知识点拆解，而是在知识点拆解之上增加一层**问题驱动的组织方式**：

```
复杂问题/项目
    ↓ 拆解
知识点A → 知识点B → 知识点C
    ↓ 各自七维拆解
HTML-A    HTML-B    HTML-C
    ↓ 组装
项目式学习路径（串联所有知识点的HTML）
```

### 两种拆解模式

| 模式 | 输入 | 输出 | 示例 |
|------|------|------|------|
| **知识点模式** | 单个知识点名称 | 1个七维HTML | "负数" → 负数.html |
| **项目式模式** | 复杂问题/项目描述 | 学习路径 + 多个七维HTML | "为什么桥梁不会塌" → 项目HTML + 多个知识点HTML |

### 项目式拆解流程

```
用户输入复杂问题/项目
    ↓
Step P1: 问题分析
  - 识别问题涉及的核心概念
  - 判断涉及的学科和范式族
  - 确定学段
    ↓
Step P2: 知识点分解
  - 将问题分解为3-7个关键知识点
  - 确定知识点之间的依赖关系（学习顺序）
  - 形成学习路径（线性/分支/网状）
    ↓
Step P3: 知识图谱感知
  - 读取 knowledge-graph.json
  - 标记已生成的知识点（复用，不重复生成）
  - 标记未生成的知识点（需要新生成）
    ↓
Step P4: 逐个生成
  - 对未生成的知识点，按知识点模式生成七维HTML
  - 每生成一个，更新 knowledge-graph.json
    ↓
Step P5: 组装学习路径
  - 生成项目式学习路径HTML
  - 包含：问题概述、学习路径图、各知识点入口
  - 已生成的知识点 → 可点击跳转
  - 未完全覆盖的知识点 → 标注"延伸学习"
    ↓
Step P6: 更新知识图谱
  - 将项目作为特殊节点写入 knowledge-graph.json
  - 项目节点与知识点节点之间用 "contains" 边关联
```

### 项目式学习路径HTML结构

```
┌─────────────────────────────────────┐
│  知本 · 为什么桥梁不会塌             │
│  [项目式学习] [物理+数学] [八年级]    │
├─────────────────────────────────────┤
│  问题概述                            │
│  桥梁承受各种力而不倒塌，背后涉及...   │
├─────────────────────────────────────┤
│  学习路径图（SVG）                   │
│                                     │
│  [力的平衡] → [力的合成与分解]        │

├─────────────────────────────────────┤
│  知识点卡片列表                      │
│  ┌──────────┐ ┌──────────┐          │
│  │ 力的平衡  │ │ 杠杆原理  │          │
│  │ 已生成 ✓  │ │ 已生成 ✓  │          │
│  │ [查看→]   │ │ [查看→]   │          │
```

## 三、知识图谱感知设计

### 核心问题

模型在生成新知识点时，必须知道：
1. **哪些知识点已经存在**（避免重复生成）
2. **已有知识点的关联关系**（复用已有链接）
3. **哪些关联节点需要更新**（新知识点生成后，旧HTML中的pending节点应变为可点击）

### 解决方案：知识图谱感知步骤

在Skill工作流程的**Step 1之后、Step 5之前**，增加一个**知识图谱感知步骤**：

```
Step 1: 识别输入
    ↓
Step 1.5: 知识图谱感知（新增）
  - 读取 knowledge-graph.json
  - 检查当前知识点是否已存在
    - 已存在 → 提示用户"已生成，是否重新生成？"
    - 不存在 → 继续
  - 提取当前知识点的关联知识中，哪些已生成、哪些未生成
  - 将已生成列表传递给后续步骤，用于：
    - Step 6（知识网络）：已生成的关联知识渲染为链接
    - Step 8（输出HTML）：知识地图中的链接正确渲染
    ↓
Step 2-5: 正常生成
    ↓
Step 7: 更新知识图谱
  - 写入新知识点
  - 更新关联边
  ↓
Step 7.5: 更新已有HTML（新增）
  - 找到所有引用了当前知识点的旧HTML
  - 将旧HTML中当前知识点的 pending 状态更新为 linked 状态
```

### 具体实现：Step 7.5 更新已有HTML

当生成新知识点X后，需要更新所有在知识地图中引用了X的旧HTML：

1. 遍历 knowledge-graph.json 的 edges，找到所有 `to: "X"` 或 `from: "X"` 的边
2. 对每条边的另一端知识点Y，检查Y的HTML文件是否存在
3. 如果存在，在Y的HTML中找到 `<span class="map-node pending">X</span>`，替换为 `<a href="X.html" class="map-node linked">X</a>`
4. 同样在知识网络维度中，找到 `X` 的pending引用，替换为链接

### 知识图谱作为"目录"

knowledge-graph.json 本身就是知识目录。模型在 Step 1.5 读取它，就能知道：
- 所有已生成的知识点列表
- 每个知识点的学科、学段、范式族
- 知识点之间的关联关系

不需要额外的目录文件。knowledge-graph.json 就是单一真相来源（Single Source of Truth）。

## 四、模式识别：如何区分知识点模式和项目式模式

### 判断规则

| 输入特征 | 模式 | 示例 |
|---------|------|------|
| 单个概念名称 | 知识点模式 | "负数""惯性""一般现在时" |
| 包含"为什么""怎么""如何"的问题 | 项目式模式 | "为什么桥梁不会塌""怎么计算圆的面积" |
| 包含多个概念的描述 | 项目式模式 | "光的折射和反射有什么关系" |
| 明确说"项目""问题""任务" | 项目式模式 | "做一个关于力学的项目" |

### Skill描述更新

在Skill的description中增加触发词：

```
Invoke when user asks to:
- decompose/拆解/讲解 a knowledge point → 知识点模式
- analyze/分析 a problem or question → 项目式模式
- do project-based learning / 项目式学习 → 项目式模式
- mentions 知本/知其然知其所以然 → 知识点模式
```

## 五、知识图谱全局视图

### 功能

- 展示所有已生成知识点的完整图谱
- 使用SVG绘制节点和连线
- 节点按学科分色
- 节点可点击跳转到对应HTML
- 项目节点用特殊样式区分

### 文件

`output/knowledge-graph.html`

### 生成时机

- 每次更新 knowledge-graph.json 后自动重新生成
- 用户也可以手动要求生成
