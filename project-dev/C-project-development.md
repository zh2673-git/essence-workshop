# 场景C：开发新项目（正向设计）

## 两种执行模式

| 模式 | 流程 | 适用场景 |
|------|------|---------|
| **标准模式** | 设计→实现→交付 | 需求明确，一次交付 |
| **迭代模式** | 设计→实现→测试→评估→保留/回退→下一轮 | 需求渐进，测试驱动，借鉴autoresearch Ratchet Loop |

---

## 两种产品开发路径

本 Skill 支持开发两类产品。两种路径都与标准/迭代模式自由组合：

| 路径 | 产物形态 | 关键差异 | 判断依据 |
|------|---------|---------|---------|
| **纯工具产品** | 无 LLM 运行时的规则/代码/数据系统 | 按现有流程设计数据结构、状态机、接口契约 | 需求结构化、无自然语言理解需求、追求确定性和低成本 |
| **协同产品** | 含 LLM 运行时 + 符号系统约束的系统 | 在现有流程基础上，额外设计 LLM 接入层、符号校验层、Prompt 管理和失败降级 | 需求模糊、需自然语言理解、需动态生成、创新性任务 |

**重要澄清**：
- 两种路径都由调用本 Skill 的 Agent（含 LLM）执行开发流程。
- 区别**不在于 Skill 是否调用 LLM**，而在于**被开发产品本身是否内置 LLM 运行时和符号系统约束**。
- 下文标准模式流程以纯工具产品为默认路径，协同产品路径的额外要求在对应步骤中用 `【协同产品补充】` 标注。

## 标准模式执行流程（坡度化）

```
第1步：是什么（坡度阶段1→2→3）
    ├── 阶段1：点出概念
    │   ├── 一句话定义（类比理解）
    │   └── 本质特征表格（从问题推导系统根本）
    ├── 阶段2：建立联系
    │   ├── 关键属性表格（从本质推导具体组成，标记✅❌）
    │   └── 区分属性（明确不是什么）
    └── 阶段3：详细讲解
        └── 设计哲学 + 对❌属性展开说明
           ↓
第2步：为什么（坡度阶段1→2→3）
    ├── 阶段1：点出概念
    │   └── 存在理由一句话总结
    ├── 阶段2：建立联系
    │   └── 技术选型表格（选型→推导逻辑）
    └── 阶段3：详细讲解
        └── 架构设计理由 + 设计模式选择
           ↓
第3步：怎么做（坡度阶段1→2→3）
    ├── 阶段1：点出概念
    │   └── 系统架构一句话概述
    ├── 阶段2：建立联系
    │   ├── 模块划分表格（标记✅❌）
    │   └── 接口契约表格
    ├── 阶段3：详细讲解
    │   ├── 全局状态设计
    │   ├── 每个模块的四层设计（数据规矩→存储→流转→接口）
    │   ├── 方法设计（状态流转图）
    │   ├── 路由设计（多场景分发）
    │   └── 可运行代码实现
    └── 验证契约执行
        ├── 定义验证契约（前置P + 后置Q + 不变量I）
        ├── validate_pre()：环境就绪？基线锁定？
        ├── validate_post()：功能正确？指标达标？
        ├── validate_inv()：旧功能不坏？接口不破？
        ├── 【协同产品补充】validate_llm()：LLM 输出可解析？符号校验通过率？幻觉检出率？降级成功率？
        └── 三维（或四维）全过 → 交付；任一失败 → 修复后重新验证
           ↓
    输出：可运行的代码实现 + 设计文档 + 验证契约
```

## 失败模式编码

> **设计依据**：借鉴 Darwin Skill dim3（失败模式编码）+ dim4（检查点显性标记）。只写正向流程的Skill是不完整的。

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| 第1步 是什么 | 核心问题定义不清，一句话定义模糊 | 追问用户"这个系统解决什么问题"，缩小范围 |
| 第1步 是什么 | 所有属性都是❌（无需深入） | 检查拆解粒度，可能需要从不同维度拆解 |
| 第2步 为什么 | 技术选型缺乏推导逻辑 | 标注"选型待验证"，提供2-3个备选方案 |
| 第2步 为什么 | 架构决策互相矛盾 | 保留矛盾，标注"架构张力"，分模块处理 |
| 第3步 怎么做 | 模块接口无法对齐 | 回到第2步重新审视架构决策 |
| 第3步 怎么做 | 全局状态设计冲突 | 重新梳理状态归属，确保单一状态源 |
| 第3步 怎么做 | 代码无法运行 | 回退到设计文档，检查接口契约是否正确 |
| 第3步 怎么做 | 验证契约前置条件不满足 | 阻断交付，修复环境/依赖后再验证 |
| 第3步 怎么做 | 验证契约后置条件不满足 | 回退到代码实现，修复功能问题 |
| 第3步 怎么做 | 验证契约不变量被破坏 | 回退到设计文档，检查接口契约是否被违反 |
| 递归深入 | 四层设计发现循环依赖 | 打破循环：提取共享部分为独立模块 |

## 检查点

- 🔴 **CHECKPOINT · 第1步完成后**：确认一句话定义和本质特征已产出，若定义模糊则追问用户
- 🔴 **CHECKPOINT · 第2步完成后**：确认技术选型有推导逻辑，架构决策无矛盾
- 🔵 **AUTO-CHECK · 第3步模块划分后**：输出模块划分和接口契约摘要，自动继续深入（用户可随时打断）
- 🔴 **CHECKPOINT · 每个模块四层设计完成后**：确认数据规矩→存储→流转→接口四层完整
- 🔴 **CHECKPOINT · 验证契约执行后**：确认P+Q+I三维验证全部通过，任一失败则修复后重新验证

## 完整示例：任务管理系统

### 第1步：是什么 (What) - 定义本质

**核心问题**：这个系统本质上是什么？

#### 1.1 本质特征（内涵）

| 推导要素 | 逻辑分析 | 结论 |
|---------|---------|------|
| **核心问题** | 用户面临什么问题？ | 任务太多，容易遗漏；任务分散，难以追踪 |
| **目标用户** | 为谁解决？ | 个人用户、小型团队 |
| **核心价值** | 解决后带来什么？ | 提高效率、减少遗漏、清晰进度 |
| **一句话定义** | 类比理解 | "这是一个任务管家，帮你记住该做的事并跟踪进度" |

**本质特征**：
1. **记忆辅助** - 系统必须记住用户创建的所有任务
2. **状态追踪** - 系统必须能追踪任务从创建到完成的全生命周期
3. **分类组织** - 系统必须支持按不同维度组织任务
4. **提醒通知** - 系统必须在关键时刻提醒用户

#### 1.2 关键属性（外延）

```
本质特征 "记忆辅助"  → 属性：任务模块（Task）✅ 需深入
本质特征 "状态追踪"  → 属性：用户模块（User）、历史模块（History）
本质特征 "分类组织"  → 属性：标签模块（Tag）、项目模块（Project）
本质特征 "提醒通知"  → 属性：调度模块（Scheduler）、通知模块（Notification）
```

| 属性名 | 推导来源 | 是否需深入 |
|-------|---------|----------|
| **任务模块** | 记忆辅助的核心 | ✅ 需深入 |
| **用户模块** | 状态追踪需要身份 | ❌ |
| **标签模块** | 分类组织需要 | ❌ |
| **项目模块** | 分类组织需要 | ❌ |
| **历史模块** | 状态追踪需要审计 | ❌ |
| **调度模块** | 提醒通知需要触发 | ❌ |
| **通知模块** | 提醒通知需要渠道 | ❌ |

#### 1.3 区分属性（边界）

- ✅ 做：个人任务管理、简单分配、状态追踪、提醒通知
- ❌ 不做：复杂日历视图、甘特图、审批工作流、资源管理

#### 1.4 设计哲学

**简单优先，快速记录，清晰追踪**

### 第2步：为什么 (Why) - 确定架构

#### 2.1 存在理由

```
因：用户大脑记忆有限，任务散落在各处，不知道优先做什么
果：提供统一入口记录任务，支持多维度分类，清晰展示状态
```

#### 2.2 技术栈决策

| 决策项 | 选择 | 推导逻辑 |
|-------|------|---------|
| **架构** | 单体应用 | 个人工具，非高并发 |
| **语言** | Python | 开发效率高，适合快速迭代 |
| **Web框架** | FastAPI | 异步支持好，类型安全 |
| **数据库** | PostgreSQL | 关系型数据适合任务关联 |

【协同产品补充】若选择协同产品路径，技术栈决策需额外增加：

| 决策项 | 选择 | 推导逻辑 |
|-------|------|---------|
| **LLM Provider** | OpenAI 兼容 API / Ollama 本地模型 | 根据成本、隐私、性能要求选择 |
| **模型路由策略** | 强模型用于复杂生成，轻量模型用于简单校验 | 平衡成本与效果 |
| **符号校验框架** | 自研或轻量规则引擎 | 与 LLM 输出格式对齐 |
| **Prompt 版本管理** | 文件化 + 版本号 | 便于追踪和回退 |
| **失败降级策略** | 规则兜底 / 人工确认 | LLM 不可用时系统仍能运行 |

### 第3步：怎么做 (How) - 实现交付

#### 3.1 全局状态设计（项目级单一状态源）

```python
from typing import TypedDict, Optional, List
from datetime import datetime

class AppState(TypedDict):
    """项目级全局状态 - 所有模块共享"""
    current_user_id: Optional[str]
    current_task_id: Optional[str]
    messages: List[dict]
    errors: List[dict]
```

**规则**：所有模块通过接口契约读写全局状态，禁止直接操作其他模块内部数据。

#### 3.2 模块划分与接口契约

```
项目（任务管理系统）
├── 用户模块（User）
│   ├── 内部状态：user_id, username, email, settings
│   └── 对外接口：register(), login(), get_settings()
│
├── 任务模块（Task）✅ 需深入
│   ├── 内部状态：task_id, title, description, status, priority, due_date
│   └── 对外接口：create(), get(), list(), transition_status()
│
├── 标签模块（Tag）
│   ├── 内部状态：tag_id, name, color
│   └── 对外接口：create(), link_to_task(), list_by_task()
│
└── ...其他模块
```

【协同产品补充】若选择协同产品路径，模块划分需显式增加以下两层：

```
├── LLM 接入层（LLM Layer）
│   ├── Provider 抽象：统一封装 OpenAI / Ollama 等接口
│   ├── 模型路由：根据任务选择强模型/轻量模型
│   ├── Prompt 管理：版本化 Prompt 模板和上下文注入
│   └── 对外接口：generate_structured(prompt, schema) → validated_output
│
└── 符号系统层（Symbol Layer）
    ├── 规则库：硬性约束 / 建议性约束 / 审计规则
    ├── 校验器：输入/输出契约、不变量、事实溯源
    ├── 反模式库：常见 LLM 失败模式
    └── 对外接口：validate(artifact, context) → ValidationResult[]
```

**协同产品接口契约设计原则**：
- LLM 节点的输入/输出必须能被符号系统解析。
- 所有 LLM 输出必须经过符号校验，失败时触发降级或重试。
- Prompt 模板、校验规则、反模式库都需版本化管理。

#### 3.3 任务模块四层设计（✅ 需深入模块示例）

**模块内部状态（局部单一状态源）**：

```python
from enum import Enum
from dataclasses import dataclass
from datetime import datetime
from typing import Optional, List

class TaskStatus(str, Enum):
    TODO = "todo"
    IN_PROGRESS = "in_progress"
    PAUSED = "paused"
    COMPLETED = "completed"
    CANCELLED = "cancelled"

@dataclass
class TaskState:
    """任务模块内部状态 - 模块级单一状态源"""
    task_id: str
    user_id: str
    title: str
    description: str = ""
    status: TaskStatus = TaskStatus.TODO
    priority: str = "medium"
    due_date: Optional[datetime] = None
    project_id: Optional[str] = None
    tag_ids: List[str] = None
    created_at: datetime = None
    updated_at: datetime = None
    started_at: Optional[datetime] = None
    completed_at: Optional[datetime] = None
    pause_reason: Optional[str] = None
    cancel_reason: Optional[str] = None
```

**四层设计**：

| 层面 | 设计内容 |
|-----|---------|
| **数据规矩** | `TaskState` Dataclass，字段类型/约束/默认值已定义；`TaskStatus` 枚举限制状态值 |
| **数据存储** | PostgreSQL `tasks` 表；异步写入；配合 History 表实现审计 |
| **数据流转** | 状态机：`todo → in_progress → completed`；流转前验证权限；失败回滚 |
| **接口层** | `TaskService` Protocol（见下方）；输入/输出/错误契约已定义 |

**模块对外接口契约**：

```python
from typing import Protocol, List, Optional
from dataclasses import dataclass

@dataclass
class TaskCreateRequest:
    title: str
    description: str = ""
    priority: str = "medium"
    due_date: Optional[datetime] = None
    tag_ids: List[str] = None
    project_id: Optional[str] = None

@dataclass
class TaskResponse:
    task_id: str
    title: str
    status: str
    priority: str
    due_date: Optional[datetime]

class TaskService(Protocol):
    """任务模块对外接口 - 外部只能通过此接口访问任务数据"""
    
    def create_task(self, user_id: str, request: TaskCreateRequest) -> TaskResponse: ...
    def get_task(self, task_id: str) -> Optional[TaskResponse]: ...
    def list_tasks(self, user_id: str, status: Optional[str] = None) -> List[TaskResponse]: ...
    def transition_status(self, task_id: str, action: str, **kwargs) -> TaskResponse: ...
```

**关键规则**：
- 外部模块（如 Notification、History）只能通过 `TaskService` 接口访问任务数据
- 禁止直接操作 `TaskState` 内部字段
- 模块内部状态变更由模块自己管理，外部无感知

## 迭代模式

迭代模式在标准模式基础上加入 Ratchet Loop 循环：设计→实现→测试→评估→保留/回退→下一轮。

**完整流程、三文件架构、评估决策、收敛判断、经验库等详见**：[iterative-loop.md](iterative-loop.md)

### 与标准模式的关系

```
标准模式（第1-2步）──→ 迭代模式（Phase 0）──→ 复用设计文档
标准模式（第3步）  ──→ 迭代模式（Phase 2）──→ 复用实现流程
迭代模式（Phase 4）──→ 蒸馏场景（B2）    ──→ 沉淀为Skill
```

迭代模式不是替代标准模式，而是在标准模式的基础上加入 Ratchet Loop 循环。
