# 协同产品 · 设计模板

> 本模板用于 `project-dev` 子 Skill 的**协同产品路径**：被开发系统本身包含 LLM 运行时 + 符号系统约束。
>
> 调用本 Skill 的 Agent 本身含有 LLM，但本模板约束的是**被开发产品**的架构设计，而非 Agent 的执行方式。

---

## 1. 路径判断

在「是什么」阶段结束时，必须明确本产品是否走协同产品路径。

| 判断维度 | 纯工具产品 | 协同产品 |
|---------|-----------|---------|
| **核心输入** | 结构化数据、明确指令 | 自然语言、模糊需求、需理解语义 |
| **核心输出** | 确定性结果 | 生成性结果，但需经符号校验 |
| **是否需要 LLM 运行时** | 否 | 是 |
| **失败处理** | 规则兜底 | LLM 输出经符号校验后规则兜底 |
| **典型例子** | 任务管理系统、数据管道 | 智能客服、AI 写作助手、代码生成 Agent |

## 3. LLM 接入层设计

### 3.1 Provider 抽象

```python
from typing import Protocol, Dict, Any

class LLMProvider(Protocol):
    def generate(self, prompt: str, context: Dict[str, Any]) -> Dict[str, Any]:
        """返回结构化输出：reasoning + content + metadata"""
        ...
```

### 3.2 模型路由策略

| 任务类型 | 推荐模型 | 说明 |
|---------|---------|------|
| 需求理解/复杂生成 | 强模型 | 多步推理、长上下文 |
| 简单格式化/校验前处理 | 轻量模型 | 成本低、延迟低 |
| 本地/隐私敏感 | Ollama 本地模型 | 数据不出本地 |

### 3.3 Prompt 管理

- Prompt 模板文件化，放在 `prompts/` 目录下。
- 每个 Prompt 文件包含版本号、适用场景、输入 schema、输出 schema、约束列表。
- Prompt 变更需走版本控制，避免线上行为漂移。

### 3.4 输出结构化

LLM 输出必须能被符号系统解析。常用格式：

- JSON Schema
- Markdown 表格
- 带标签的文本（如 `<reasoning>...</reasoning><output>...</output>`）

## 4. 符号系统层设计

### 4.1 规则库

| 规则级别 | 说明 | 是否阻塞 |
|---------|------|---------|
| **hard** | 硬性约束，违反即失败 | 是 |
| **soft** | 建议性约束，违反记录为 warning | 否 |
| **audit** | 仅记录，不反馈给 LLM | 否 |

### 4.2 校验器

```python
from dataclasses import dataclass
from typing import List, Dict, Any

@dataclass
class ValidationResult:
    status: str           # "pass" | "fail" | "warning"
    rule_id: str
    severity: str         # "error" | "warning" | "info"
    location: str
    message: str
    suggestion: str
    anti_pattern: str = ""

class SymbolValidator:
    def validate(self, artifact: Any, context: Dict[str, Any]) -> List[ValidationResult]:
        ...
```

### 4.3 反模式库

记录 LLM 常见错误模式，用于 Prompt 上下文注入和自动校验。

| 反模式 ID | 名称 | 症状 |
|----------|------|------|
| AP-001 | 输出不可解析 | LLM 未按指定格式输出 |
| AP-002 | 事实捏造 | 引用不存在的数据或来源 |
| AP-003 | 风格漂移 | 输出偏离指定风格 |
| AP-004 | 忽视边界条件 | 未处理空值、异常、边界 |
| AP-005 | 违反接口契约 | LLM 输出字段与契约不一致 |

## 5. 模块四层设计（以 LLM 节点为例）

对于协同产品，每个 LLM 节点都需要独立的四层设计。

| 层面 | 设计内容 | 示例 |
|-----|---------|------|
| **数据规矩** | 输入/输出 Schema、字段类型、默认值、约束 | `GenerateRequest` / `GenerateResponse` Dataclass |
| **数据存储** | Prompt 模板、历史对话、校验日志的存储位置 | `prompts/v1/task_prompt.txt`、`logs/validation.log` |
| **数据流转** | 用户输入 → Prompt 组装 → LLM 调用 → 符号校验 → 业务模块 | 状态机或 Pipeline |
| **接口层** | LLM 节点的对外接口契约 | `generate_structured(request: GenerateRequest) -> GenerateResponse` |

## 6. 验证契约（协同产品专属）

在标准 P/Q/I 三维验证基础上，增加 `validate_llm()` 维度。

### 6.1 前置条件 P

| 条件 | 含义 | 验证方式 |
|------|------|---------|
| 环境就绪 | 依赖已安装、LLM API 可访问 | 运行环境检查脚本 |
| Prompt 模板就绪 | Prompt 文件存在且版本正确 | 文件检查 |
| 符号规则就绪 | 规则库已加载 | 规则加载测试 |

### 6.2 后置条件 Q

| 条件 | 含义 | 验证方式 |
|------|------|---------|
| 功能正确 | LLM 节点按预期工作 | 测试用例通过 |
| 符号校验通过率达标 | ≥ 目标阈值（如 95%） | 批量测试 |
| 降级机制有效 | LLM 失败时能规则兜底 | 注入失败测试 |

### 6.3 不变量 I

| 条件 | 含义 | 验证方式 |
|------|------|---------|
| 旧功能不坏 | 已有测试仍然通过 | 回归测试 |
| 接口不破 | 对外接口契约不变 | 接口兼容性检查 |
| Prompt 不回退 | Prompt 版本变更需显式记录 | 版本控制 diff |

### 6.4 LLM 专属指标 validate_llm()

| 指标 | 含义 | 目标值 |
|------|------|--------|
| **可解析率** | LLM 输出能被符号系统解析的比例 | ≥ 98% |
| **校验通过率** | 解析后通过符号校验的比例 | ≥ 95% |
| **幻觉检出率** | 符号系统成功拦截错误/捏造的比例 | ≥ 90% |
| **降级成功率** | LLM 失败时规则兜底成功的比例 | ≥ 99% |

## 7. 失败模式编码

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| LLM 调用 | 输出无法解析 | 记录反模式；重试；连续失败则降级 |
| 符号校验 | 硬性约束失败 | 返回结构化反馈；LLM 修正；超次则人工确认 |
| 符号校验 | 事实捏造 | 拦截输出；要求 LLM 提供来源；无法提供则降级 |
| 模型路由 | LLM Provider 不可用 | 切换到备用 Provider；无备用则规则兜底 |
| Prompt 版本 | Prompt 变更导致行为漂移 | 版本回退；A/B 测试验证新 Prompt |
| 降级机制 | 规则兜底也失败 | 返回友好错误；记录日志；人工介入 |
