# 场景A2：K12知识拆解（知本）

> 把知识点从"结论"还原为"认知过程"，让学习者重走发现之路。

> **方法论锚点**：本场景采用 [../../cognitive-engine/general-logic/references/methodology.md](../../cognitive-engine/general-logic/references/methodology.md) —— 本质定义先行 + 顿悟触发机制 + 类-属性-方法-路由由本质定义锚定。在七维框架展开前，必须先为知识点找到本质定义并通过本质检验；七个维度都必须能回溯到本质定义。

## 触发条件

- 用户说「拆解XX」「讲解XX」「XX怎么理解」
- 用户提到「知本」「知其然知其所以然」
- 用户提到「项目式学习」「问题驱动学习」
- 用户输入单个概念名称（知识点模式）
- 用户输入包含"为什么""怎么""如何"的问题（项目式模式）
- 用户说「继续更新」「检查并补充」「增量更新」（批量增量更新模式）

## 模式判断

| 输入特征 | 模式 | 示例 |
|---------|------|------|
| 单个概念名称 | **知识点模式** | "负数""惯性""一般现在时" |
| 包含"为什么""怎么""如何"的问题 | **项目式模式** | "为什么桥梁不会塌" |
| 包含多个概念的描述 | **项目式模式** | "光的折射和反射有什么关系" |
| 明确说"项目""问题""任务" | **项目式模式** | "做一个关于力学的项目" |
| 说"继续更新""增量更新""检查补充" | **批量增量更新模式** | "继续更新知识图谱" |
| 单纯要求拆解/讲解一个概念 | **知识点模式** | "拆解一下负数" |

## 核心原则

1. **还原认知，而非传递结论**——不是告诉学生答案，而是让学生看到答案是怎么被发现的
2. **每个维度必须有具体支撑**——禁止空洞表述，故事要有细节，例子要有场景
3. **质量高于速度**——宁可少写一句，也不写一句废话
4. **本质定义先行**——在七维展开前，必须先找到本质定义并通过本质检验；本质定义要能解释这个知识点的所有其他属性
5. **顿悟触发三件套**——维度1结尾有一句话点破、维度3含反常识根因、维度5含视角升级
6. **回溯本质**——七个维度的每个内容都必须能回溯到本质定义，无法回溯的视为冗余

## 工作流程A：知识点模式

### Step 1: 识别输入

从用户输入中提取：
- **知识点名称**（必需）
- **学科**（必需，如未指定则根据知识点推断）
- **学段**（可选，默认初中）

学科映射：数学=math, 物理=physics, 化学=chemistry, 语文=chinese, 历史=history, 英语=english, 生物=biology, 地理=geography

### Step 1.2: 本质定义先行（顿悟触发①）—— 调用 cognitive-engine

> **认知分析细节**走 [../../cognitive-engine/general-logic/references/methodology.md](../../cognitive-engine/general-logic/references/methodology.md) —— 本质定义先行、本质检验、顿悟触发机制。本步骤只规定编排调用，本质定义/本质检验/类归属等方法由 cognitive-engine 执行。

**编排调用**：在七维框架展开前，先调用 cognitive-engine 执行阶段1（本质定义先行）：
1. 调用 cognitive-engine 抛本质定义（揭底，不是描述外延）
2. 调用 cognitive-engine 执行本质检验（能解释所有核心属性则通过；不能则降级重写）
3. 由本质定义确定类归属（决定后续维度展开方向）
4. 产出"一句话点破"（顿悟触发①，可独立传播，能让学习者30秒抓住核心）

**K12 适配要求**（场景层规则，非认知引擎规则）：
- 本质定义必须用学段能理解的语言（小学生用生活语言，初中生用半抽象语言，高中生可用抽象语言）
- 一句话点破要能在课堂开头讲出来引发好奇，不是百科式定义

### Step 1.5: 知识图谱感知与增量更新

**必须读取 `{输出目录}/knowledge-graph.json`**（如不存在则跳过）。

1. **检查当前知识点是否已存在**：
   - 不存在 → 继续（全新生成）
   - 已存在 → 进入**增量检查流程**：

   **增量检查流程**：
   a. 读取已有HTML文件，检查7个维度的完整性：
      - 7个维度是否全部有内容（非空、非占位符）
      - 认知组（定义锚点/历史溯源/现实矛盾）是否有具体细节
      - 定义锚点是否有生活锚点/体验锚点
      - 历史溯源是否有具体故事（人物+困境+突破）
      - 现实矛盾是否有困境时刻
      - 应用场景是否有3个不同领域
      - 验证理解是否有3题以上
      - 知识网络是否有前导和后续知识点
      - SVG图示是否已生成
   b. 根据检查结果分类：
      - **完整**（所有必检项通过）→ 提示用户"该知识点已完整生成（{html_file}），是否需要更新？"
      - **不完整**（有维度缺失或质量不达标）→ 列出缺失/不达标项，提示用户"该知识点已存在但不完整，缺失：{列表}，是否补充？"
      - 用户确认补充 → 仅重新生成缺失/不达标的维度，保留已有内容
      - 用户确认重新生成 → 按全新流程生成（覆盖旧文件）

2. **提取关联知识的生成状态**：
   - 从知识网络维度中涉及的关联知识点，检查哪些已在nodes中
   - 已生成的关联知识 → 记录为"可链接"，后续渲染为 `<a href="xxx.html">`
   - 未生成的关联知识 → 记录为"待生成"，后续渲染为 `<span class="map-node pending">`

3. **将感知结果传递给后续步骤**：Step 5（知识网络维度）、Step 8（知识地图渲染）

### Step 2: 加载范式族插件

根据学科选择范式族：

| 学科 | 范式族 | 插件文件 |
|------|--------|---------|
| 数学/物理/化学 | 形式科学族 | references/formal-science.md |
| 语文/历史/英语 | 人文学科族 | references/humanities.md |
| 生物/地理 | 生命科学族 | references/life-science.md |

**必须读取对应的插件文件**，获取该范式族各维度的生成规则。

### Step 3: 加载学段适配规则

读取 `references/grade-adapter.md`，获取对应学段的表述规则。

### Step 4: 生成SVG图示

根据范式族为关键维度生成SVG图示。图示是认知的脚手架，不是装饰。

**必须生成图示的维度**：

| 范式族 | 维度 | 图示类型 |
|--------|------|---------|
| 形式科学族 | 定义锚点 | 数轴/坐标系图（展示概念位置和边界） |
| 形式科学族 | 现实矛盾 | 逻辑困境图（展示"没有它"的断裂） |
| 人文学科族 | 定义锚点 | 情感光谱/手法定位图 |
| 人文学科族 | 历史溯源 | 时间线图（4W可视化） |
| 生命科学族 | 定义锚点 | 系统/结构示意图 |
| 生命科学族 | 现实矛盾 | 表象vs本质对比图 |

**可选图示**：应用场景的实验示意图、思维延伸的单位关系图/手法谱系图。

> **SVG/图示规范**走 [../../format-pipeline/html](../../format-pipeline/html)，由 format-pipeline 提供图示样式与容器规范。

### Step 5: 按七维框架生成内容

按以下顺序生成7个维度的内容：

**认知组（理解）**——先生成，质量必须最高：
1. 定义锚点
2. 历史溯源
3. 现实矛盾

**应用组（迁移）**——后生成：
4. 应用场景
5. 思维延伸
6. 知识网络

**验证组（确认）**——最后生成：
7. 验证理解

每个维度的生成规则来自：主框架（下方）+ 范式族插件（Step 2加载）+ 学科微调（插件文件中）+ 学段适配（Step 3加载）。

### Step 6: 质量自检

生成完成后，按以下清单自检：

**必检项（任一不通过则修正后重新生成）**：
- [ ] 7个维度全部有内容
- [ ] 认知组3个维度各有具体细节（非空洞表述）
- [ ] 定义锚点有生活锚点/体验锚点/可观察现象
- [ ] 历史溯源有具体故事（人物+困境+突破）
- [ ] 现实矛盾有困境时刻
- [ ] 验证理解有3题以上，且不是纯计算/纯背诵
- [ ] 无AI套话（"探索""旅程""赋能""深入""全面"等空洞词）

**本质锚定必检项（任一不通过则修正后重新生成）**：
- [ ] Step 1.2 本质定义已通过本质检验（能解释所有核心属性）
- [ ] 维度1结尾有"一句话点破"（顿悟触发①，可独立传播）
- [ ] 维度3含"反常识根因"（顿悟触发②，至少1个反直觉点）
- [ ] 维度5结尾有"视角升级"（顿悟触发③，不是总结）
- [ ] 七个维度的内容都能回溯到本质定义（无法回溯的已删除或重写）
- [ ] 类-属性-方法-路由由本质定义锚定（不是隐喻硬接）

**选检项（不通过则补充）**：
- [ ] 应用场景有3个不同领域
- [ ] 应用场景有1个可动手验证
- [ ] 知识网络有前导和后续知识点

### Step 7: 更新知识图谱

读取 `{输出目录}/knowledge-graph.json`（如不存在则创建），将当前知识点和关联信息写入：

```json
{
  "nodes": {
    "知识点名称": {
      "id": "kebab-case-id",
      "name": "知识点名称",
      "subject": "math",
      "phase": "初中",
      "grade": 7,
      "paradigm_family": "formal_science",
      "html_file": "{输出目录}/{学段}/{知识域或学科}/{知识点名称}.html",
      "generated_at": "日期"
    }
  },
  "edges": [
    { "from": "前导知识", "to": "当前知识", "type": "prerequisite", "label": "前导" },
    { "from": "当前知识", "to": "后续知识", "type": "subsequent", "label": "后续" }
  ]
}
```

**phase 规则**：grade 1-6 → "小学"，grade 7-9 → "初中"，grade 10-12 → "高中"。phase 用于知识域划分（学段+学科），grade 用于学段内排序。

### Step 7.5: 更新已有HTML中的知识地图

当生成新知识点X后，需要更新所有在知识地图中引用了X的旧HTML：

1. 遍历 knowledge-graph.json 的 edges，找到所有 `from: "X"` 或 `to: "X"` 的边
2. 对每条边的另一端知识点Y，检查Y的HTML文件是否存在
3. 如果存在，在Y的HTML中找到引用X的pending节点，替换为可点击链接
4. 同样更新Y的知识网络维度中对X的引用

> **链接渲染与跨目录路径计算**走 [../../format-pipeline/html](../../format-pipeline/html)，由 format-pipeline 提供 pending/linked 节点样式与相对路径计算规范。

### Step 7.6: 同步更新知识图谱全局视图

**每次生成新知识点或更新 knowledge-graph.json 后，必须同步更新 `{输出目录}/knowledge-graph.html`**：

1. 读取最新的 `{输出目录}/knowledge-graph.json`
2. 将完整数据写入 `knowledge-graph.html` 中的 `GRAPH_DATA` 变量
3. 确保 nodes 和 edges 与 JSON 文件完全一致
4. 不需要用户手动触发，这是自动步骤

### Step 8: 输出HTML

> **HTML模板与渲染规范**走 [../../format-pipeline/html](../../format-pipeline/html)（模板：[templates/decomposition.html](../../format-pipeline/html/templates/decomposition.html)）。本步骤只规定编排调用与输出路径占位符。

调用 format-pipeline/html 将7个维度的内容填入模板，生成完整HTML文件。

**输出路径规则**（{输出目录} 占位符，实际路径由 platform-adapter 决定；按学段/学科分类存放）：

| 学科 | 输出路径模板 |
|------|------------|
| 数学 | `{输出目录}/{学段}/{知识域}/{知识点名称}.html` |
| 物理 | `{输出目录}/{学段}/物理/{知识点名称}.html` |
| 语文 | `{输出目录}/{学段}/语文/{知识点名称}.html` |
| 英语 | `{输出目录}/{学段}/英语/{知识点名称}.html` |
| 信息技术 | `{输出目录}/{学段}/信息技术/{知识点名称}.html` |
| 项目式 | `{输出目录}/项目/{项目名称}.html` |

数学知识域推断规则：含"图形/几何/角/三角形/圆/面积/体积/对称/坐标"等关键词 → 图形与几何；含"统计/概率/数据" → 统计与概率；含"实践/综合" → 综合与实践；其余 → 数与代数。

学段规则：grade 1-6 → 小学，grade 7-9 → 初中，grade 10-12 → 高中。

**知识地图中的链接规则**（编排层面，渲染细节走 format-pipeline）：
- 已生成的关联知识点（在knowledge-graph.json的nodes中存在且有html_file）→ 渲染为可点击链接
- 未生成的关联知识点 → 渲染为待生成节点

### Step 9: 递归生成（可选）

用户可以要求递归生成关联知识点：

- **手动递归**：用户说"继续生成前导知识"或"继续生成后续知识"，按用户指定方向递归
- **自动递归1层**：用户说"生成完整知识链"，自动生成所有直接前导和后续（1层）
- **自动递归N层**：用户说"递归生成N层"，自动生成N层关联

递归时：
1. 从当前知识点的知识网络维度提取关联知识点
2. 检查knowledge-graph.json，跳过已生成的
3. 按Step 1-8生成每个新知识点
4. 更新所有相关HTML中的知识地图链接
5. 更新knowledge-graph.json

### Step 10: 生成知识图谱全局视图

首次生成 `{输出目录}/knowledge-graph.html`。后续每次新增知识点时，由 Step 7.6 自动同步更新，无需用户手动触发。

全局视图包含（编排要求，渲染走 format-pipeline/html）：
- 使用SVG绘制节点和连线
- 节点按学科分色
- 节点可点击跳转到对应HTML
- 显示关联类型标签
- 项目节点用特殊样式（虚线边框、更大尺寸）区分

## 工作流程B：项目式模式

当用户输入是复杂问题/项目描述时，使用此流程。

### Step P1: 问题分析

从用户输入中提取：
- **核心问题**（必需）：用户想理解什么
- **涉及学科**（必需）：判断涉及哪些学科
- **学段**（可选，默认初中）
- **问题类型**：因果解释型 / 设计制作型 / 比较分析型

### Step P2: 知识点分解

将问题分解为3-7个关键知识点，确定学习顺序：

1. **识别核心概念**：问题中直接涉及的概念
2. **补充前导概念**：理解核心概念需要的前置知识
3. **确定依赖关系**：哪些知识必须先学
4. **形成学习路径**：按依赖关系排列学习顺序

### Step P3: 知识图谱感知

**必须读取 `{输出目录}/knowledge-graph.json`**（如不存在则跳过）。

1. 检查每个分解出的知识点是否已生成
2. 已生成的 → 标记为"复用"，不重复生成
3. 未生成的 → 标记为"待生成"
4. 检查已生成知识点的关联知识是否覆盖了部分需求

### Step P4: 逐个生成

对"待生成"的知识点，按工作流程A（Step 1-8）逐个生成：
- 每生成一个，更新 knowledge-graph.json
- 每生成一个，执行 Step 7.5 更新已有HTML

### Step P5: 组装学习路径

生成项目式学习路径HTML，包含以下部分：

1. **问题概述**：用学习者能理解的语言重述问题
2. **学习路径图**（SVG）：展示知识点之间的依赖关系和学习顺序
3. **知识点卡片列表**：
   - 已生成的 → 显示"已生成"标记 + 可点击跳转
   - 未生成的 → 显示"待生成"标记
4. **综合思考**：回到原始问题，用已学知识回答
5. **延伸问题**：提出2-3个进阶问题

输出文件路径：`{输出目录}/{项目名称}.html`

使用模板：`../../format-pipeline/html/templates/project.html`

**项目节点写入知识图谱**：
```json
{
  "nodes": {
    "项目名称": {
      "id": "kebab-case-id",
      "name": "项目名称",
      "type": "project",
      "subjects": ["physics", "math"],
      "phase": "初中",
      "grade": 8,
      "html_file": "{输出目录}/项目名称.html",
      "contains": ["知识点A", "知识点B", "知识点C"],
      "generated_at": "日期"
    }
  },
  "edges": [
    { "from": "项目名称", "to": "知识点A", "type": "contains", "label": "包含" },
    { "from": "项目名称", "to": "知识点B", "type": "contains", "label": "包含" }
  ]
}
```

### Step P6: 更新知识图谱

1. 将项目节点和 contains 边写入 knowledge-graph.json
2. 执行 Step 7.5 更新已有HTML
3. 重新生成知识图谱全局视图（如已存在）

## 工作流程C：批量增量更新

当用户说"继续更新""检查并补充""增量更新"时，使用此流程。自动扫描知识图谱，优先补充不完整的已有知识点，然后生成新的待拆解知识点。

### Step I1: 扫描知识图谱状态

读取 `{输出目录}/knowledge-graph.json`，将所有知识点分为三类：

| 类别 | 条件 | 优先级 |
|------|------|--------|
| **待生成** | 在nodes中但无html_file | 最低 |
| **不完整** | 有html_file但HTML内容缺失维度 | 最高 |
| **完整** | 有html_file且7维度全部达标 | 跳过 |

### Step I2: 完整性检查（对"不完整"类）

对每个有html_file的知识点，读取其HTML文件，按以下清单检查：

**必检项**（任一不通过则标记为"不完整"）：
- [ ] 7个维度section全部存在且非空
- [ ] 认知组3个维度各有具体细节（非空洞表述）
- [ ] 定义锚点有生活锚点/体验锚点
- [ ] 历史溯源有具体故事（人物+困境+突破）
- [ ] 现实矛盾有困境时刻
- [ ] 验证理解有3题以上

**选检项**（不通过则标记为"可改进"）：
- [ ] 应用场景有3个不同领域
- [ ] 知识网络有前导和后续知识点
- [ ] SVG图示已生成

### Step I3: 生成更新计划

向用户展示扫描结果：

```
📊 知识图谱增量更新报告
- 完整：38个
- 不完整：5个（列出名称和缺失维度）
- 可改进：3个（列出名称和改进项）
- 待生成：116个

建议更新顺序：
1. 先补充不完整的5个（优先级最高）
2. 再改进3个可改进的
3. 最后生成新的待拆解知识点

是否按此顺序执行？
```

### Step I4: 逐个执行更新

按用户确认的顺序逐个处理：

1. **补充不完整知识点**：
   - 仅重新生成缺失/不达标的维度
   - 保留已有内容不变
   - 更新HTML文件（替换对应section）
   - 不需要重新生成整个文件

2. **改进可改进知识点**：
   - 补充缺失的选检项内容
   - 保留已有内容不变

3. **生成新知识点**：
   - 按工作流程A（Step 1-8）完整生成
   - 优先选择关联度高的知识点（与已有知识点有prerequisite/subsequent关系的）

每处理完一个知识点：
- 更新 knowledge-graph.json
- 执行 Step 7.5 更新关联HTML
- 执行 Step 7.6 同步知识图谱全局视图
- 向用户报告进度

### Step I5: 进度追踪

在 knowledge-graph.json 中维护更新进度：

```json
{
  "metadata": {
    "last_incremental_check": "日期",
    "total_nodes": 154,
    "complete": 38,
    "incomplete": 5,
    "improvable": 3,
    "pending": 116,
    "last_updated_node": "知识点名称"
  }
}
```

下次执行增量更新时，从 `last_updated_node` 之后继续，避免重复检查。

## 七维主框架（编排结构）

> **认知规则细节**走 [../../cognitive-engine/general-logic/references/methodology.md](../../cognitive-engine/general-logic/references/methodology.md) —— 本质定义先行、本质检验、因果链构建、反常识根因提取、视角升级、类-属性-方法-路由锚定等方法由 cognitive-engine 提供。本节只规定七个维度的编排顺序与维度职责，每个维度的认知分析细节走 cognitive-engine。
>
> **锚定要求**：七个维度的每个内容都必须能回溯到 Step 1.2 的本质定义。无法回溯的视为冗余，应删除或重写。本质定义像一根线，把七个维度串成一条认知链而非七个孤立模块。

| 维度 | 本质问题 | 编排职责（调用 cognitive-engine 的哪些能力） |
|------|---------|-------------------------------------------|
| **维度1：定义锚点（本质锚点）** | 它是什么？ | 调用阶段1（本质定义+本质检验+类归属+一句话点破·顿悟触发①）；用生活锚点/体验锚点锚定概念边界 |
| **维度2：历史溯源** | 它从哪来？ | 调用因果链构建（果→因反推）；讲关键发现节点（人物+困境+突破），突破点必须回溯到本质定义 |
| **维度3：现实矛盾（顿悟触发②）** | 为什么需要它？ | 调用反常识根因提取（顿悟触发②）；展示"没有它会怎样"的困境，反常识根因必须由本质定义推导 |
| **维度4：应用场景** | 它能做什么？ | 由本质定义推导应用（"因为本质是X，所以可用于场景Y"）；至少3个不同领域例子 |
| **维度5：思维延伸（顿悟触发③）** | 它还能怎样？ | 调用视角升级（顿悟触发③）；链接后续知识点，提出进阶问题，重新定义问题给出更高视角 |
| **维度6：知识网络** | 它和什么关联？ | 由本质定义推导关联类型（前导/后续/平行/跨学科）；至少1个前导+1个后续知识点 |
| **维度7：验证理解** | 真的懂了吗？ | 检验本质理解（不考计算考理解，不考背诵考感受）；至少1题考"本质定义"，至少3题 |

> 每个维度的具体认知分析方法（如何抛本质定义、如何做本质检验、如何构建因果链、如何提取反常识根因、如何做视角升级等）详见 cognitive-engine/general-logic/references/methodology.md。范式族对各维度的微调规则见 references/ 下的范式族插件文件。

## 禁止事项

1. 禁止AI套话：不要用"探索""旅程""赋能""深入""全面""让我们一起"等空洞词
2. 禁止空洞表述：每个观点必须有具体支撑
3. 禁止Emoji
4. 禁止紫色渐变
5. 禁止纯学术定义没有锚点
6. 禁止只有年代名字没有故事
7. 禁止纯计算/纯背诵的验证题
8. **禁止**：用描述性定义代替本质定义（必须通过本质检验）
9. **禁止**：视角升级退化为总结（禁用"综上所述""由此可见""总的来说"等套话）
10. **禁止**：维度内容无法回溯到本质定义（应删除或重写）
11. **禁止**：因果链顺向无反差（必须有至少1个反常识根因）

## 工作流程C：讲课模式（项目式视频录制）

当用户要求录制项目式学习视频，或说"讲课模式""自动讲解""录制视频"时使用。

### 问题

项目式学习涉及多个HTML文件（项目HTML + 多个知识点HTML），视频录制时无法展示跨页面跳转。

### 方案：讲课导航页

生成一个**单页讲课导航HTML**，将项目概述和所有知识点的核心内容内联到一个页面中，模拟"老师讲课"的流程：

1. 项目概述（问题 + 学习路径）
2. 知识点1（完整七维内容，内联）
3. 知识点2（完整七维内容，内联）
4. ...
5. 综合思考
6. 回到项目问题

每个知识点之间有"过渡页"（类似PPT翻页），讲课模式自动逐段滚动+展开。

### Step L1: 收集知识点

1. 读取 `{输出目录}/knowledge-graph.json`
2. 找到项目节点，提取 contains 列表
3. 检查每个知识点的HTML是否已生成
4. 全部已生成 → 继续；有未生成的 → 先按工作流程A生成

### Step L2: 生成讲课导航页

> **HTML模板与演示脚本**走 [../../format-pipeline/html](../../format-pipeline/html)（模板：[templates/lecture.html](../../format-pipeline/html/templates/lecture.html)）。本步骤只规定编排结构与内容组装顺序。

调用 format-pipeline/html，读取所有知识点的HTML，提取七维内容，按以下结构组装为单页HTML：

```
讲课导航页结构：
├── 封面（项目问题 + 学科标签）
├── 学习路径图（SVG，带进度指示）
├── 知识点1
│   ├── 过渡页（"接下来我们学习：XXX"）
│   ├── 七维内容（全部展开，无折叠）
│   └── 小结（一句话总结）
├── 知识点2
│   ├── 过渡页
│   ├── 七维内容
│   └── 小结
├── ...
├── 综合思考（回到原始问题）
└── 延伸问题
```

输出文件路径：`{输出目录}/{项目名称}-lecture.html`

### Step L3: 内置讲课模式演示

> **演示脚本（分段播放/进度指示/自动滚动/段间暂停/URL参数/键盘控制等）**走 [../../format-pipeline/html](../../format-pipeline/html)，由 format-pipeline 提供讲课模式的交互逻辑实现。

### Step L4: 视频管线集成

> **视频录制管线**走 [../../format-pipeline/video](../../format-pipeline/video)。

讲课导航页可直接用于视频管线录制：

1. 打开 `{项目名称}-lecture.html?lecture=1`
2. 调用 format-pipeline/video 录制整个页面的自动演示过程
3. 无需手动操作，无需跨页面跳转
4. 录制完成后得到完整的项目式学习视频
