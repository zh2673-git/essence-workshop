# 场景D：解析现有项目（逆向分析）

## 思考层 vs 输出层分离原则

> **核心修正**：方法论（逆向三阶合一、本质定义先行、顿悟触发）是**思考层脚手架**，决定反推的深度和因果链的严密性。产物的外形由**反推报告的固有结构**决定（实现观察→设计推导→本质反推→改进建议），**禁止把思考结构硬塞到产物分节中**。
>
> **类比**：考古学家用碳14测年法确定文物年代（思考层），但博物馆展陈按历史叙事组织（输出层）。不能因为用了碳14法，就让展柜按碳14衰变曲线分章。
>
> **两层桥接**：
> - **思考层**（必走，不强制露出）：逆向三阶合一（实现观察→设计推导→本质反推）、本质检验、顿悟触发作为智能体内部反推与质量校验
> - **输出层**（按反推报告结构组织）：产物按反推报告固有结构组织；顿悟触发点融入产物，位置由体裁决定；本质反推结果可在"本质总结"章节露出，但**不强制要求产物开头抛本质**
> - **思考工具不硬塞**：本质检验表、"回溯到本质"列、"## 阶段3/阶段2/阶段1"分节标题、逆向三阶的思考分节等思考工具不暴露在产物中
>
> **顿悟触发在反推报告中的位置映射**：
> | 顿悟触发 | 思考层位置 | 输出层位置（反推报告体裁） |
> |---------|---------|-----------------------|
> | 视角升级（实现层） | 逆向第1步思考结尾 | 实现观察章节的"实现洞察"小节 |
> | 反常识根因（设计层） | 逆向第2步思考中段 | 设计推导章节的"设计意图分析"小节 |
> | 一句话点破（本质层） | 逆向第3步思考结尾 | 本质总结章节的"本质洞察"小节（不必开头抛出） |
> | 本质定义 | 逆向第3步思考 | 是思考根锚，可在"本质总结"章节露出，不强制开头 |
> | 本质检验 | 逆向第3步思考 | 是思考质量校验，**不在产物里露出** |

## 思考层流程：逆向三阶合一编排（智能体内部推理脚手架）

> **认知分析细节**走 [../../cognitive-engine/general-logic/references/methodology.md](../../cognitive-engine/general-logic/references/methodology.md) —— 逆向三阶合一、本质定义先行、本质检验、顿悟触发机制、类-属性-方法-路由由本质定义锚定。本场景只规定编排调用顺序，认知分析步骤（本质定义/本质检验/因果链反推/反常识根因提取/视角升级等方法）的执行细节由 cognitive-engine 提供。
>
> **方法论对齐**：场景D 是逆向推导——阶段3（怎么做·观察实现）← 阶段2（为什么·反推因果）← 阶段1（是什么·反推本质）。
> 从成品反推本质定义：先观察实现（阶段3），再反推因果（阶段2），最后反推本质定义并做本质检验（阶段1）。
> 顿悟触发在逆向场景下顺序倒置：第1步结尾"视角升级（实现层）"→ 第2步中段"反常识根因（设计层）"→ 第3步结尾"一句话点破（本质层）"。
> 类-属性-方法-路由由反推出的本质定义锚定：任意属性/方法/路由如无法回溯到反推出的本质，说明反推不完整，需回到上游补充观察。
>
> **分离提醒**：以下逆向三阶是**思考层脚手架**，决定反推的深度。智能体走完此流程后，**产物按"输出层结构"小节中定义的反推报告固有结构组织**（实现观察→设计推导→本质反推→改进建议），不按"## 阶段3/阶段2/阶段1"分节。

## 执行流程

```
第1步：调用阶段3（怎么做·观察实现·视角升级）— 逆向推导的入口
    ├── 1.1 获取项目信息 → GitHub元数据、README
    ├── 1.2 浏览源码结构 → 目录组织、文件分布
    ├── 1.3 识别模块划分 → 从代码结构识别属性（标记 ✅需深入 的模块）
    ├── 1.4 分析方法实现 → 方法如何利用属性（推断方法的本质驱动）
    └── 1.5 由 cognitive-engine 执行视角升级（顿悟触发·实现层）
        **遇到看不懂的代码时**：参考 [references/code-reading-guide.md](references/code-reading-guide.md)
           ↓
第2步：调用阶段2（为什么·反推因果·反常识根因）
    ├── 2.1 由 cognitive-engine 反推因果链（果→因）
    ├── 2.2 技术选型分析 → 从源码反推决策原因
    ├── 2.3 架构设计推导 → 从结构反推设计思想
    └── 2.4 由 cognitive-engine 提取反常识根因（顿悟触发·设计层）
           ↓
第3步：调用阶段1（是什么·反推本质·本质定义先行）
    ├── 3.1 由 cognitive-engine 抛本质定义（反推，揭底不是描述外延）
    ├── 3.2 由 cognitive-engine 执行本质检验（反推出的本质能否解释所有观察到的属性/方法/路由）
    │   ├── 能解释 → 反推成功，保留
    │   └── 不能解释 → 反推不完整，回到第1步/第2步补充观察
    ├── 3.3 类归属与属性确认 → 由本质定义确定类归属；验证属性由本质展开；由本质界定系统边界
    └── 3.4 由 cognitive-engine 产出一句话点破（顿悟触发·本质层）
           ↓
    输出：项目反推报告（00-{repo}-反推报告.md，按反推报告固有结构组织，不按阶段分节；平台由 platform-adapter 决定）
           ↓
第4步：递归深入 - 分析标记模块（自动执行）
    **触发条件**：项目级文档中存在 ✅需深入 标记的模块

    对每个 ✅需深入 的模块，按序号顺序执行：
    ├── 4.1 读取该模块的源码文件
    ├── 4.2 调用 cognitive-engine 执行逆向三阶分析（怎么做→为什么→是什么）
    │   **递归纪律**：递归只重做阶段1（重新找子概念的本质定义），
    │   不再重复阶段2/3 的完整流程，避免无限展开
    ├── 4.3 输出四层基础设施分析
    ├── 4.4 生成模块级反推报告（01-{模块A}-反推报告.md）
    ├── 4.5 检查该模块是否还有 ✅需深入 的子模块
    │   └── 如有，递归执行第4步（序号递增：02-, 03-...）
    └── 4.6 当前模块所有 ✅ 分析完毕，返回上一级
           ↓
    所有模块递归分析完毕
           ↓
    最终输出：完整的项目文档集合
```

### 顿悟触发与本质锚定（思考层强制要求，输出层按反推报告体裁融入）

> **修正**：顿悟触发点是**内容要求**（必须存在于产物中），不是**分节要求**（位置由体裁决定）。顿悟触发方法（如何提炼视角升级、如何提取反常识根因、如何做一句话点破）由 cognitive-engine 提供。

| 触发点 | 触发器 | 思考层位置（逆向顺序） | 输出层位置（反推报告体裁） | 验收标准 |
|-------|-------|--------------------|-----------------------|---------|
| 第1步结尾 | 视角升级（实现层） | 1.5 思考结尾 | **实现观察**章节的"实现洞察"小节 | 看穿"实现为什么这么写"，高于纯描述 |
| 第2步中段 | 反常识根因（设计层） | 2.4 思考中段 | **设计推导**章节的"设计意图分析"小节 | 反推出至少1个反直觉设计点，并由反推出的本质解释 |
| 第3步结尾 | 一句话点破（本质层） | 3.4 思考结尾 | **本质总结**章节的"本质洞察"小节（不必开头抛出） | 反推出的本质的最简洁表达，可独立传播 |

**类-属性-方法-路由由反推出的本质定义锚定**：每一层必须能回溯到第3步反推出的本质定义。
- 类 ← 由反推出的本质定义确定归属
- 属性 ← 必须是反推出本质的展开
- 方法 ← 必须是反推出本质的能动方式
- 路由 ← 必须是反推出本质的情境适配

任意属性/方法/路由如果无法回溯到反推出的本质定义，说明反推不完整——需回到第1步/第2步补充观察。

**思考工具（不硬塞到产物中）**：本质检验表、"回溯到本质"列、"## 阶段3/阶段2/阶段1"分节标题等思考工具不暴露在产物中。

## 失败模式编码

> **设计依据**：借鉴 Darwin Skill dim3（失败模式编码）+ dim4（检查点显性标记）。只写正向流程的Skill是不完整的。
> **修正**：区分**思考层失败**（影响反推深度，不直接体现为产物失败）和**产物层失败**（影响交付质量）。思考工具缺失不应等同于产物失败。

### 思考层失败（影响反推深度，需重做思考流程）

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| 第1步 怎么做 | 视角升级退化为描述实现 | 重写，要求看穿"实现为什么这么写"，高于纯描述 |
| 第2步 为什么 | 反推因果链起点无法回溯到任何候选本质 | 标注"因果链起点待定"，进入第3步后用本质检验回查 |
| 第2步 为什么 | 设计意图无法从代码推导 | 标注"设计意图不明"，提供2-3种可能解释 |
| 第2步 为什么 | 技术选型原因不可知 | 标注"选型原因推测"，不强行确定 |
| 第2步 为什么 | 找不到反常识根因 | 检查是否有反直觉设计点；如全为顺向设计，标注"无明显反常识点"，不强求 |
| 第3步 是什么 | 反推出的本质定义通不过本质检验（不能解释所有观察到的属性） | 反推不完整，回到第1步/第2步补充观察，再重抛本质定义 |
| 第3步 是什么 | 本质特征推导出多个矛盾结论 | 保留多个视角，标注"存在多种解读"；逐个做本质检验，保留能解释最多的 |
| 第3步 是什么 | 类归属靠"项目=类"隐喻硬接 | 由反推出的本质重新确定类归属，不靠隐喻 |
| 第3步 是什么 | 一句话点破不可独立传播 | 重写，去除修饰，留下反推出本质的最简表达 |
| 第3步 是什么 | 系统边界模糊 | 标注"边界待确认"，列出由反推本质界定的可能边界 |

### 产物层失败（影响交付质量，按交付项分类）

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| 第1步 怎么做 | GitHub仓库无法访问 | 尝试本地克隆；若仍失败则询问用户是否有本地副本 |
| 第1步 怎么做 | 项目无README/文档 | 从源码结构推断，标注"无文档，推断可能不准" |
| 第1步 怎么做 | 代码量过大无法全读 | 优先读入口文件+核心模块，其余标注"未深入" |
| 第4步 递归深入 | 模块源码缺失/被混淆 | 标注"源码不可读"，跳过该模块 |
| 第4步 递归深入 | 递归超过3层仍未终止 | 强制终止（递归只重做阶段1，不重复阶段2/3，避免无限展开），标注"深入不足" |
| 产物组织 | 产物按"## 阶段3/阶段2/阶段1"分节而非反推报告结构 | 重组为反推报告固有结构（实现观察→设计推导→本质反推→改进建议） |

## 检查点

> **修正**：检查点区分思考层（智能体内部校验，不直接产出文档）和输出层（产物质量校验）。

**思考层检查点**（智能体内部校验，不直接产出文档）：
- 🔴 **CHECKPOINT · 第1步思考完成后**：确认项目结构和模块划分已识别，关键模块已标记✅；视角升级（实现层）已产出
- 🔴 **CHECKPOINT · 第2步思考完成后**：确认反推因果链已完成，反常识根因（设计层）已产出；无法推导的部分已标注
- 🔴 **CHECKPOINT · 第3步思考完成后**：确认本质定义已反推并通过本质检验；类归属已由本质锚定；一句话点破已产出

**输出层检查点**（产物质量校验）：
- 🔵 **AUTO-CHECK · 第3步思考完成后**：输出本质反推结果摘要（含本质定义+一句话点破，本质检验不露出产物），自动继续递归深入（用户可随时打断）
- 🔴 **CHECKPOINT · 每个模块递归分析后**：确认四层分析完整，无遗漏
- 🔴 **CHECKPOINT · 产物组织完成后**：确认产物按反推报告固有结构组织（不按阶段分节），顿悟触发点已融入对应章节

## 检查清单（两层）

> **修正**：检查清单分两层。思考层检查走方法论但不在产物露出；输出层检查按反推报告结构。

### 思考层检查（智能体内部校验，不在产物中露出）

- [ ] 第1步思考：已识别项目结构和模块划分，关键模块已标记✅
- [ ] 第1步思考结尾：已产出视角升级（实现层）——看穿"实现为什么这么写"
- [ ] 第2步思考：已反推因果链（果→因），起点能回溯到候选本质
- [ ] 第2步思考中段：已产出反常识根因（设计层）——至少1个反直觉设计点（若无，已标注不强求）
- [ ] 第3步思考：已反推出本质定义（揭底，不是描述外延）
- [ ] 第3步思考：已通过本质检验（反推出的本质能解释所有观察到的属性/方法/路由）
- [ ] 第3步思考：已确定类归属（由反推出的本质锚定，不靠"项目=类"隐喻）
- [ ] 第3步思考结尾：已产出一句话点破（本质层）——反推出本质的最简洁表达
- [ ] 产物组织：**未按"## 阶段3/阶段2/阶段1"分节，按反推报告固有结构组织**
- [ ] 产物组织：本质检验表、"回溯到本质"列等思考工具未硬塞到产物中

### 输出层检查（按反推报告结构组织，产物质量校验）

- [ ] 已获取GitHub项目元数据
- [ ] 已浏览项目目录结构
- [ ] 已识别核心模块和入口文件
- [ ] 已执行逆向三阶思考分析（思考层走，产物层不按此分节）
- [ ] 已生成项目级反推报告文档（`00-{repo}-反推报告.md`）
- [ ] 已标记需深入分析的模块（✅）
- [ ] 已完成基础设施四层分析
- [ ] 实现观察章节：包含项目结构、模块划分、实现洞察（融入"视角升级"）
- [ ] 设计推导章节：包含反推因果链、设计意图分析（融入"反常识根因"）
- [ ] 本质反推章节：包含本质定义、本质总结、本质洞察（融入"一句话点破"），不必开头抛出
- [ ] 改进建议章节（可选）：基于反推提出改进方向

**模块级分析（递归）**
- [ ] 对每个 ✅需深入 的模块执行深入分析
- [ ] 递归只重做阶段1（重新找子概念的本质定义），不重复阶段2/3完整流程
- [ ] 已生成模块级文档
- [ ] 模块文档包含四层分析
- [ ] 没有更多 ✅需深入 的模块时停止

**汇总输出**
- [ ] 已生成架构图谱（99-架构图谱.md）
- [ ] 已更新项目索引（00-GitHub项目索引.md）

**质量保障**
- [ ] 失败模式：每个关键步骤的失败分支已编码
- [ ] 检查点：🔵 AUTO-CHECK 检查点已输出摘要

## 输出层结构：反推报告（按场景固有结构组织）

> **文档格式与模板**走 [../../format-pipeline/markdown](../../format-pipeline/markdown) 与本场景 [templates/project-analysis.md](templates/project-analysis.md)。本场景只规定文件编排结构（哪些文件、什么命名、什么递归关系），具体文档模板由 format-pipeline 与 templates 提供。
>
> **修正**：产物按反推报告的固有结构组织，**不按"## 阶段3/阶段2/阶段1"分节**。逆向三阶在思考层走，决定反推的深度，不决定产物外形。

```
{输出目录}/
├── {org}-{repo}/
│   ├── 00-{repo}-反推报告.md      # 项目级反推报告
```

### 反推报告文档（00-{repo}-反推报告.md）的固有结构

> 反推报告按**固有体裁结构**组织（项目概览→实现观察→设计推导→本质反推→改进建议），不是思考层逆向三阶的分节。具体模板见 [templates/project-analysis.md](templates/project-analysis.md)。顿悟触发点（视角升级/反常识根因/一句话点破）融入对应章节，位置由体裁决定。

## 文档命名规范

| 文档类型 | 命名格式 | 编排职责（调用哪些引擎/模板） |
|---------|---------|------|
| **项目级反推报告** | `00-{repo}-反推报告.md` | 调用逆向三阶完整流程；项目概览+实现观察+设计推导+本质反推+改进建议 |
| **模块级反推报告** | `{序号}-{模块名}-反推报告.md` | 调用逆向三阶（递归只重做阶段1）；模块级四层分析 |
| **架构图谱** | `99-架构图谱.md` | 汇总架构关系图谱 |
| **项目索引** | `00-GitHub项目索引.md` | 所有项目索引汇总 |

> **不再硬塞的思考工具**：本质检验表、"回溯到本质"列、"## 阶段3/阶段2/阶段1"分节标题等思考工具不暴露在产物中。
>
> **历史命名兼容**：原 `00-{repo}-三阶解构.md` 命名保留为历史快照可读；新产出统一采用 `00-{repo}-反推报告.md` 命名。
