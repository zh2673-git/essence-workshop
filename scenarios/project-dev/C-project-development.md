# 场景C：开发新项目（正向设计）

## 思考层 vs 输出层分离原则

> **核心修正**：方法论（三阶合一、本质定义先行、顿悟触发、类-属性-方法-路由）是**思考层脚手架**，决定设计的深度和决策依据。产物的外形由**项目方案的固有结构**决定（本质定义→架构总览→模块递归→接口契约→验证方式），**禁止把思考结构硬塞到产物分节中**。
>
> **类比**：盖房子时结构工程师用力学公式算承重（思考层），但房子外形由建筑师设计（输出层）。不能因为用了力学公式，就让房子长得像公式。
>
> **两层桥接**：
> - **思考层**（必走，不强制露出）：三阶合一、本质定义先行、本质检验、顿悟触发作为智能体内部推理与设计质量校验
> - **输出层**（按项目方案结构组织）：产物按项目方案固有结构组织；顿悟触发点融入产物，位置由体裁决定；类-属性-方法-路由在项目方案中**是合适的输出结构**，可以露出
> - **思考工具不硬塞**：本质检验表、"回溯到本质"列、"## 阶段1是什么 / ## 阶段2为什么 / ## 阶段3怎么做"分节等思考工具不暴露在产物中
>
> **顿悟触发在项目方案中的位置映射**：
> | 顿悟触发 | 思考层位置 | 输出层位置（项目方案体裁） |
> |---------|---------|-----------------------|
> | 一句话点破 | 阶段1结尾 | 项目目标 / 项目愿景 |
> | 反常识根因 | 阶段2中段 | 设计依据 / 技术选型理由 |
> | 视角升级 | 阶段3结尾 | 项目愿景 / 未来扩展 |
> | 本质定义 | 阶段1 | 是思考根锚，可在"项目目标"或"本质定义"章节露出 |
> | 本质检验 | 阶段1 | 是思考质量校验，**不在产物里露出** |
> | 类-属性-方法-路由 | 阶段3 | 项目方案中**是合适的输出结构，可露出** |

## 两种执行模式

| 模式 | 流程 | 适用场景 |
|------|------|---------|
| **标准模式** | 设计→实现→交付 | 需求明确，一次交付 |
| **迭代模式** | 设计→实现→测试→评估→保留/回退→下一轮 | 需求渐进，测试驱动，借鉴autoresearch Ratchet Loop |

## 两种产品开发路径

本 Skill 支持开发两类产品。两种路径都与标准/迭代模式自由组合：

| 路径 | 产物形态 | 关键差异 | 判断依据 |
|------|---------|---------|---------|
| **纯工具产品** | 无 LLM 运行时的规则/代码/数据系统 | 按现有流程设计数据结构、状态机、接口契约 | 需求结构化、无自然语言理解需求、追求确定性和低成本 |
| **协同产品** | 含 LLM 运行时 + 符号系统约束的系统 | 在现有流程基础上，额外设计 LLM 接入层、符号校验层、Prompt 管理和失败降级 | 需求模糊、需自然语言理解、需动态生成、创新性任务 |

**重要澄清**：
- 两种路径都由调用本 Skill 的 Agent（含 LLM）执行开发流程。
- 区别**不在于 Skill 是否调用 LLM**，而在于**被开发产品本身是否内置 LLM 运行时和符号系统约束**。
- 下文标准模式流程以纯工具产品为默认路径，协同产品路径的额外要求在对应步骤中用 `【协同产品补充】` 标注。

## 思考层流程：三阶合一编排（智能体内部推理脚手架）

> **认知分析细节**走 [../../cognitive-engine/general-logic/references/methodology.md](../../cognitive-engine/general-logic/references/methodology.md) —— 三阶合一、本质定义先行、本质检验、顿悟触发机制、类-属性-方法-路由由本质定义锚定。本场景只规定编排调用顺序，认知分析步骤（本质定义/本质检验/因果链构建/反常识根因提取/视角升级等方法）的执行细节由 cognitive-engine 提供。
>
> **分离提醒**：以下三阶合一是**思考层脚手架**，决定设计的深度和决策依据。智能体走完此流程后，**产物按"输出层结构"小节中定义的项目方案固有结构组织**，不按"## 阶段1/阶段2/阶段3"分节。
>
> 三阶与坡度三阶段一一对应，不再正交为9宫格。每阶只走一次，避免重复。
> 每一阶的产物必须能回溯到阶段1的本质定义（类-属性-方法-路由由本质定义锚定）。

```
第1步：调用阶段1（是什么·本质定义先行）
    └── 由 cognitive-engine 执行：本质定义 → 本质检验 → 类归属 → 关键属性展开 → 区分属性 → 一句话点破（顿悟触发①）
           ↓
第2步：调用阶段2（为什么·建立因果）
    └── 由 cognitive-engine 执行：因果链 → 技术选型推导 → 架构设计理由 → 反常识根因（顿悟触发②）→ 结构性比喻
           ↓
第3步：调用阶段3（怎么做·展开方法·视角升级）
    ├── 由 cognitive-engine 执行：模块划分 → 方法设计（本质在动作中的体现）→ 路由设计（本质的情境适配）→ 视角升级（顿悟触发③）
    ├── 每个模块的四层设计（数据规矩→存储→流转→接口）
    ├── 可运行代码实现
    └── 验证契约执行（前置P + 后置Q + 不变量I；协同产品额外 validate_llm()）
           ↓
    输出：可运行的代码实现 + 设计文档 + 验证契约
```

### 顿悟触发与本质锚定（思考层强制要求，输出层按项目方案体裁融入）

> 顿悟触发点是**内容要求**（必须存在于产物中），不是**分节要求**（位置由体裁决定）。顿悟触发方法（如何提炼一句话点破、如何提取反常识根因、如何做视角升级）由 cognitive-engine 提供。

| 触发点 | 触发器 | 思考层位置 | 输出层位置（项目方案体裁） | 验收标准 |
|-------|-------|---------|-----------------------|---------|
| 阶段1结尾 | 一句话点破 | 第1步思考末尾 | **项目目标** / 项目愿景章节 | 最简洁本质表达，可独立传播 |
| 阶段2中段 | 至少1个反常识根因 | 第2步因果链思考中 | **设计依据** / 技术选型理由章节 | 反直觉点，制造认知冲突 |
| 阶段3结尾 | 视角升级 | 第3步思考收尾 | **项目愿景** / 未来扩展章节 | 重新定义问题，高于阶段1视角 |

**类-属性-方法-路由由本质定义锚定**（在项目方案中是合适的输出结构，可以露出）：
- 类 ← 由本质定义确定归属
- 属性 ← 必须是本质的展开
- 方法 ← 必须是本质的能动方式
- 路由 ← 必须是本质的情境适配

任意属性/方法/路由如果无法回溯到本质定义，即为冗余或错位。

**思考工具（不硬塞到产物中）**：本质检验表、"回溯到本质"列等是思考质量校验工具，不在产物里露出。

## 输出层结构：项目方案（按场景固有结构组织）

> **文档格式与模板**走 [../../format-pipeline/markdown](../../format-pipeline/markdown) 与本场景 [templates/](templates/) 目录（项目方案/架构/模块/四层设计等模板）。本场景只规定文件编排结构（哪些文件、什么命名、什么递归关系），具体文档模板由 format-pipeline 与 templates 提供。

> 产物按项目方案的固有结构组织，**不按"## 阶段1是什么 / ## 阶段2为什么 / ## 阶段3怎么做"分节**。三阶合一在思考层走，决定设计深度，不决定产物外形。

### 项目方案文档（01-项目方案.md）的固有结构

> 项目方案按**固有体裁结构**组织（项目目标→本质定义→架构总览→模块递归→验证方式→项目愿景），不是思考层三阶的分节。具体模板见 [templates/project-essence.md](templates/project-essence.md)、[templates/project-architecture.md](templates/project-architecture.md)、[templates/project-modules.md](templates/project-modules.md)、[templates/four-layer-design.md](templates/four-layer-design.md)。顿悟触发点（一句话点破/反常识根因/视角升级）融入对应章节，位置由体裁决定。

## 失败模式编码

> **设计依据**：借鉴 Darwin Skill dim3（失败模式编码）+ dim4（检查点显性标记）。只写正向流程的Skill是不完整的。
> 区分**思考层失败**（影响设计质量，不直接体现为产物失败）和**产物层失败**（影响交付质量）。思考工具缺失不应等同于产物失败。

### 思考层失败（影响设计深度，需重做思考流程）

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| 第1步 是什么 | 本质定义通不过本质检验（不能解释所有其他属性） | 降级重写，找到能解释所有其他属性的本质定义 |
| 第1步 是什么 | 核心问题定义不清，本质定义模糊 | 追问用户"这个系统解决什么问题"，缩小范围后重抛本质定义 |
| 第1步 是什么 | 类归属靠"项目=类"隐喻硬接 | 由本质定义重新确定类归属，不靠隐喻 |
| 第1步 是什么 | 一句话点破不可独立传播 | 重写，去除修饰，留下最简本质表达 |
| 第1步 是什么 | 所有属性都是❌（无需深入） | 检查拆解粒度，可能需要从不同维度拆解 |
| 第2步 为什么 | 因果链起点不能回溯到本质定义 | 重新推导因果链，确保起点来自阶段1本质 |
| 第2步 为什么 | 找不到反常识根因 | 检查因果链是否线性顺向，强行制造认知冲突点；若确实无反直觉点，不强求 |
| 第2步 为什么 | 技术选型缺乏推导逻辑 | 标注"选型待验证"，提供2-3个备选方案 |
| 第2步 为什么 | 架构决策互相矛盾 | 保留矛盾，标注"架构张力"，分模块处理 |
| 第3步 怎么做 | 方法/路由无法回溯到本质定义 | 判定为冗余或错位，剔除或重写 |
| 第3步 怎么做 | 视角升级退化为总结 | 重写，要求"重新定义问题"，给出比阶段1更高的视角 |

### 产物层失败（影响交付质量，按交付项分类）

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| 第3步 怎么做 | 模块接口无法对齐 | 回到第2步重新审视架构决策 |
| 第3步 怎么做 | 全局状态设计冲突 | 重新梳理状态归属，确保单一状态源 |
| 第3步 怎么做 | 代码无法运行 | 回退到设计文档，检查接口契约是否正确 |
| 第3步 怎么做 | 验证契约前置条件不满足 | 阻断交付，修复环境/依赖后再验证 |
| 第3步 怎么做 | 验证契约后置条件不满足 | 回退到代码实现，修复功能问题 |
| 第3步 怎么做 | 验证契约不变量被破坏 | 回退到设计文档，检查接口契约是否被违反 |
| 递归深入 | 四层设计发现循环依赖 | 打破循环：提取共享部分为独立模块 |
| 递归深入 | 递归超过3层仍未终止 | 强制终止（递归只重做阶段1，不重复阶段2/3，避免无限展开） |
| 产物组织 | 产物按"## 阶段1/2/3"分节而非项目方案结构 | 重组为项目方案固有结构（项目目标→本质定义→架构总览→模块递归→验证方式→项目愿景） |

## 检查点

> 检查点区分思考层（智能体内部校验，不直接产出文档）和输出层（产物质量校验）。

**思考层检查点**（智能体内部校验，不直接产出文档）：
- 🔴 **CHECKPOINT · 第1步思考完成后**：确认本质定义已通过本质检验、类归属已由本质定义锚定、一句话点破已产出；若定义模糊则追问用户
- 🔴 **CHECKPOINT · 第2步思考完成后**：确认因果链起点回溯到本质定义、至少1个反常识根因已产出、技术选型有推导逻辑、架构决策无矛盾
- 🔴 **CHECKPOINT · 第3步思考收尾时**：确认视角升级已产出（重新定义问题，非总结）

**输出层检查点**（产物质量校验）：
- 🔵 **AUTO-CHECK · 第3步模块划分后**：输出模块划分和接口契约摘要，自动继续深入（用户可随时打断）
- 🔴 **CHECKPOINT · 每个模块四层设计完成后**：确认数据规矩→存储→流转→接口四层完整
- 🔴 **CHECKPOINT · 验证契约执行后**：确认P+Q+I三维验证全部通过，任一失败则修复后重新验证
- 🔴 **CHECKPOINT · 产物组织完成后**：确认产物按项目方案固有结构组织（不按阶段分节），顿悟触发点已融入对应章节

## 检查清单（两层）

> 检查清单分两层。思考层检查不在产物露出；输出层检查按项目方案结构。

### 思考层检查（智能体内部校验，不在产物中露出）

- [ ] 阶段1：已抛出本质定义（不是描述性定义，是揭底）
- [ ] 阶段1：已通过本质检验（能解释所有其他属性吗？）
- [ ] 阶段1：已确定类归属（由本质定义锚定，不靠隐喻硬接）
- [ ] 阶段1：已从本质定义推导出关键属性，并标记 ✅需深入 的模块
- [ ] 阶段1：已产出一句话点破（顿悟触发①）
- [ ] 阶段1：已明确系统边界（由本质定义界定不做什么）
- [ ] 阶段2：因果链起点已回溯到阶段1的本质定义
- [ ] 阶段2：已产出至少1个反常识根因（顿悟触发②）
- [ ] 阶段2：已推导技术选型理由和架构决策（由本质推导）
- [ ] 阶段3：方法设计已回溯到本质（本质在动作中的体现）
- [ ] 阶段3：路由设计已回溯到本质（本质的情境化适配）
- [ ] 阶段3：已产出视角升级收尾（顿悟触发③，重新定义问题）
- [ ] 产物组织：**未按"## 阶段1/2/3"分节，按项目方案固有结构组织**
- [ ] 产物组织：本质检验表、"回溯到本质"列等思考工具未硬塞到产物中

### 输出层检查（按项目方案结构组织，产物质量校验）

- [ ] 项目方案文档（`01-项目方案.md`）：包含项目目标、本质定义、架构总览、模块递归、验证方式、项目愿景
- [ ] 项目目标章节：已融入"一句话点破"（最简洁本质表达，可独立传播）

## 完整示例：任务管理系统

> **说明**：本示例以"第1步/第2步/第3步"分节呈现，是为了**演示思考层三阶流程怎么走**。实际产出的项目方案文档应按"输出层结构"小节中定义的**项目方案固有结构**组织（项目目标→本质定义→架构总览→模块递归→验证方式→项目愿景），**不要把示例中的"第1步/第2步/第3步"分节直接搬到产物里**。
>
> 示例中的顿悟触发点（一句话点破、反常识根因、视角升级）的位置只是思考层位置，最终在产物中按项目方案体裁融入对应章节。

### 第1步：是什么 (What) - 本质定义先行（思考层流程）

**核心问题**：这个系统本质上是什么？

#### 1.1 本质定义（揭底，不是描述外延）

> **本质定义**：任务管理系统是用**显性化的状态机**换取**大脑的解放**——把"该做什么、做到哪了、何时做"从生物记忆迁移到外部状态机，让大脑专注于执行而非记忆与检索。

对比（说明为什么不能停留在描述性定义）：

| 类型 | 形式 | 例子 | 能否锚定类-属性-方法-路由 |
|-----|------|------|------------------------|
| 描述性定义 | 属加种差，描述外延 | "任务管理系统是记录任务并跟踪状态的软件" | ❌ 只说"长什么样" |
| **本质定义** | 揭示本质属性 | "用外部状态机换大脑解放" | ✅ 决定所有其他属性 |

#### 1.2 本质检验（阶段1必须通过）

本质定义必须能解释这个东西的所有其他属性——能解释 → 保留；不能解释 → 降级重写。

| 属性 | 候选定义（描述性）能否解释 | 本质定义（状态机换大脑）能否解释 | 说明 |
|------|------------------------|---------------------------------|------|
| 记忆辅助 | ❌（"记录任务"≠ 必然记住） | ✅（状态机持久化是记忆的本质） | 描述性降级 |
| 状态追踪 | ❌（记录 ≠ 必然有状态机） | ✅（状态机即状态流转） | 描述性降级 |
| 分类组织 | ❌ | ✅（分类是把任务从大脑的混沌映射到状态机的有序结构） | 描述性降级 |
| 提醒通知 | ❌ | ✅（提醒是状态机驱动的触发器） | 描述性降级 |

**结论**：本质定义通过检验，保留。

#### 1.3 类归属（由本质定义确定，不靠"项目=类"隐喻硬接）

由"用外部状态机换大脑解放"这一本质定义，确定任务管理系统属于 **外部状态机类**（而非"任务软件类"这种描述性归类）。

#### 1.4 关键属性展开（由本质定义推导）

```
本质定义 "状态机换大脑解放"
   ├── "状态机" → 属性：任务模块（Task，状态载体）✅ 需深入
   ├── "持久化" → 属性：历史模块（History，状态审计）
   ├── "状态归属" → 属性：用户模块（User，状态所有者）
   ├── "有序结构" → 属性：标签模块（Tag）、项目模块（Project）
   └── "状态触发器" → 属性：调度模块（Scheduler）、通知模块（Notification）
```

| 属性名 | 推导来源（回溯到本质） | 是否需深入 |
|-------|----------------------|----------|
| **任务模块** | 状态机的状态载体（本质核心） | ✅ 需深入 |
| **用户模块** | 状态归属（状态机必须有所有者） | ❌ |
| **标签模块** | 有序结构（状态机的结构化能力） | ❌ |
| **项目模块** | 有序结构（状态机的结构化能力） | ❌ |
| **历史模块** | 持久化与审计（状态机的可回溯性） | ❌ |
| **调度模块** | 状态触发器（状态机的主动驱动） | ❌ |
| **通知模块** | 状态触发器（状态机的外部响应） | ❌ |

#### 1.5 区分属性（由本质定义界定系统边界）

- ✅ 做：任务状态机管理、简单分配、状态流转追踪、状态驱动提醒（这些都回溯到"状态机换大脑解放"）
- ❌ 不做：复杂日历视图、甘特图、审批工作流、资源管理（这些都不依赖外部状态机的核心价值）

#### 1.6 一句话点破（顿悟触发①·最简洁的本质表达）

> **任务管理系统的本质不是"管理任务"，而是"用状态机替你记忆"——你解放出来的是大脑，不是时间。**

### 第2步：为什么 (Why) - 建立因果

#### 2.1 因果链（起点必须回溯到阶段1的本质定义）

本质定义是"用外部状态机换大脑解放"，由此推导因果：

| 因 | 果 | 逻辑 | 反差/类比 |
|----|----|------|---------|
| 大脑记忆容量有限、易遗漏 | 任务需要被外部持久化 | 状态机的持久化能力 | 反差：人脑擅长联想不擅长枚举 |
| 任务分散在多入口（笔记/聊天/邮件） | 必须有统一的状态归属 | 状态机必须有单一状态源 | 类比：水库集中蓄水 vs 各家挖井 |
| 状态机要驱动动作，不能静止 | 需要状态触发器（调度/通知） | 状态机的主动驱动能力 | 反差：被动提醒 vs 主动驱动 |
| 状态机要可审计 | 历史模块记录状态变更 | 状态机的可回溯性 | 类比：飞行黑匣子 |

#### 2.2 反常识根因（顿悟触发②·至少1个反直觉点）

> **反常识根因**：好的任务管理系统，**不在于功能多**，而在于**减少用户与状态机之间的摩擦**。
>
> 直觉认为"功能越丰富越能管理好任务"，但本质定义告诉我们：状态机换大脑解放，前提是状态机的输入输出**不增加大脑负担**。如果录入任务比记住任务更累，状态机就被绕过了——所以"少输入、多自动驱动"才是设计核心，而不是堆功能。
>
> 这解释了为什么 Trello/Things 这类极简工具比"功能堆砌型"任务管理软件更受欢迎——它们让状态机对用户的摩擦更小。

#### 2.3 技术栈决策（由本质推导，杜绝随意选型）

| 决策项 | 选择 | 推导逻辑（回溯本质） |
|-------|------|----------------------|
| **架构** | 单体应用 | "状态机换大脑"是个人工具的本质，单体足以承载单一状态机 |
| **语言** | Python | 状态机的状态流转需要类型安全与开发效率的平衡 |
| **Web框架** | FastAPI | 异步支持状态机的触发器（调度/通知） |
| **数据库** | PostgreSQL | 关系型数据适合状态机的多维度查询与状态归属 |

【协同产品补充】若选择协同产品路径，技术栈决策需额外增加：

| 决策项 | 选择 | 推导逻辑 |
|-------|------|---------|
| **LLM Provider** | OpenAI 兼容 API / Ollama 本地模型 | 根据成本、隐私、性能要求选择 |
| **模型路由策略** | 强模型用于复杂生成，轻量模型用于简单校验 | 平衡成本与效果 |
| **符号校验框架** | 自研或轻量规则引擎 | 与 LLM 输出格式对齐 |
| **Prompt 版本管理** | 文件化 + 版本号 | 便于追踪和回退 |
| **失败降级策略** | 规则兜底 / 人工确认 | LLM 不可用时系统仍能运行 |

#### 2.4 结构性比喻（可选，比喻内部结构与本质同构）

> **结构性比喻**：任务管理系统像"外置硬盘"——大脑是 RAM（快但易失），状态机是硬盘（慢但持久）。
> 内部结构同构：RAM 的"读写"对应任务的"创建/查询"；硬盘的"持久化"对应历史模块的审计；硬盘的"索引"对应标签/项目的分类；硬盘的"定时备份"对应调度/通知的触发。比喻的每个部分都能映射回本质。

### 第3步：怎么做 (How) - 展开方法·视角升级

#### 3.1 全局状态设计（项目级单一状态源）

> 全局状态定义项目级共享状态（如当前用户、当前任务、消息、错误等），具体代码实现由开发者按语言规范编写。

**规则**：所有模块通过接口契约读写全局状态，禁止直接操作其他模块内部数据。

#### 3.2 模块划分与接口契约

```
项目（任务管理系统）
├── 用户模块（User）
│   ├── 内部状态：user_id, username, email, settings
│   └── 对外接口：register(), login(), get_settings()
│
├── 任务模块（Task）✅ 需深入
│   ├── 内部状态：task_id, title, description, status, priority, due_date
│   └── 对外接口：create(), get(), list(), transition_status()
│
├── 标签模块（Tag）
│   ├── 内部状态：tag_id, name, color
│   └── 对外接口：create(), link_to_task(), list_by_task()
│
└── ...其他模块
```

【协同产品补充】若选择协同产品路径，模块划分需显式增加以下两层：

```
├── LLM 接入层（LLM Layer）
│   ├── Provider 抽象：统一封装 OpenAI / Ollama 等接口
│   ├── 模型路由：根据任务选择强模型/轻量模型
│   ├── Prompt 管理：版本化 Prompt 模板和上下文注入
│   └── 对外接口：generate_structured(prompt, schema) → validated_output
│
└── 符号系统层（Symbol Layer）
    ├── 规则库：硬性约束 / 建议性约束 / 审计规则
    ├── 校验器：输入/输出契约、不变量、事实溯源
    ├── 反模式库：常见 LLM 失败模式
    └── 对外接口：validate(artifact, context) → ValidationResult[]
```

**协同产品接口契约设计原则**：
- LLM 节点的输入/输出必须能被符号系统解析。
- 所有 LLM 输出必须经过符号校验，失败时触发降级或重试。
- Prompt 模板、校验规则、反模式库都需版本化管理。

#### 3.3 任务模块四层设计（✅ 需深入模块示例）

**模块内部状态（局部单一状态源）**：

> 模块内部状态定义任务的状态载体（如 task_id、title、status、priority、due_date 等字段）与状态枚举（如 todo/in_progress/paused/completed/cancelled），具体代码实现由开发者按语言规范编写。

**四层设计**：

| 层面 | 设计内容 |
|-----|---------|
| **数据规矩** | 任务状态 Dataclass（字段类型/约束/默认值）+ 状态枚举（限制状态值） |
| **数据存储** | PostgreSQL `tasks` 表；异步写入；配合 History 表实现审计 |
| **数据流转** | 状态机：`todo → in_progress → completed`；流转前验证权限；失败回滚 |
| **接口层** | TaskService 接口（create/get/list/transition_status）；输入/输出/错误契约已定义 |

**模块对外接口契约**：

> 对外接口契约定义 TaskService 的方法签名（create_task、get_task、list_tasks、transition_status）及请求/响应数据结构，具体代码实现由开发者按语言规范编写。

**关键规则**：
- 外部模块（如 Notification、History）只能通过 TaskService 接口访问任务数据
- 禁止直接操作任务内部状态字段
- 模块内部状态变更由模块自己管理，外部无感知

**方法与路由的本质锚定说明**：
- **方法 = 本质在动作中的体现**：`transition_status()` 即"状态机驱动状态流转"这一本质的动作化；`create_task()` 即"外部状态机接收输入"。
- **路由 = 本质在不同场景的情境化适配**：`list_tasks(user_id, status=None)` 按用户和状态筛选，是"状态机的多维度查询"在不同用户视角下的分发。
- 任意方法/路由如果无法回溯到"状态机换大脑解放"这一本质，即为冗余或错位。

#### 3.4 视角升级（顿悟触发③·收尾，不是总结，是重新定义问题）

> **视角升级**：阶段1我们说"任务管理系统是用状态机换大脑解放"，做完实现后视角应当升级——
>
> 真正的本质不是"做一个任务管理软件"，而是**设计一个"用户大脑外延"的协议层**：状态机只是协议的当前实现，未来这个协议可以是 LLM 驱动的意图识别 + 状态机执行的混合体（协同产品路径），也可以是脑机接口直连。
>
> 这意味着：判断一个"任务管理"创新是否成立，不是看它新增了多少功能，而是看它**是否在减小用户与状态机之间的协议摩擦**。语音录入、自动归类、智能提醒——都是在改协议，不是在改状态机本身。
>
> **重新定义的问题**：不是"如何管理任务"，而是"如何最小化用户与外部状态机之间的协议成本"。下次设计类似系统时，先问"协议摩擦在哪"，而不是"功能还缺什么"。

## 迭代模式

迭代模式在标准模式基础上加入 Ratchet Loop 循环：设计→实现→测试→评估→保留/回退→下一轮。

**完整流程、三文件架构、评估决策、收敛判断、经验库等详见**：[iterative-loop.md](iterative-loop.md)

### 与标准模式的关系

```
标准模式（第1-2步）──→ 迭代模式（Phase 0）──→ 复用设计文档
标准模式（第3步）  ──→ 迭代模式（Phase 2）──→ 复用实现流程
迭代模式（Phase 4）──→ 蒸馏场景（B2）    ──→ 沉淀为Skill
```

迭代模式不是替代标准模式，而是在标准模式的基础上加入 Ratchet Loop 循环。
