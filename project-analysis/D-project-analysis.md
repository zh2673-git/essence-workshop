# 场景D：解析现有项目（逆向分析）

## 执行流程

```
第1步：怎么做 (How) - 观察实现
    ├── 1.1 获取项目信息 → GitHub元数据、README
    ├── 1.2 浏览源码结构 → 目录组织、文件分布
    ├── 1.3 识别模块划分 → 从代码结构识别属性
    │                    └── 标记 ✅需深入 的模块
    └── 1.4 分析方法实现 → 方法如何利用属性
        **遇到看不懂的代码时**：参考 [references/code-reading-guide.md](references/code-reading-guide.md)
           ↓
第2步：为什么 (Why) - 推导设计
    ├── 2.1 技术选型分析 → 从源码推导决策原因
    ├── 2.2 架构设计推导 → 从结构推导设计思想
    └── 2.3 设计亮点识别 → 可借鉴的模式
           ↓
第3步：是什么 (What) - 推导本质
    ├── 3.1 本质特征推导 → 从代码推导系统根本
    ├── 3.2 关键属性确认 → 验证识别的属性
    └── 3.3 区分属性界定 → 明确系统边界
           ↓
    输出：项目理解文档（00-{repo}-三阶解构.md 保存到Obsidian）
           ↓
第4步：递归深入 - 分析标记模块（自动执行）
    **触发条件**：项目级文档中存在 ✅需深入 标记的模块
    
    对每个 ✅需深入 的模块，按序号顺序执行：
    ├── 4.1 读取该模块的源码文件
    ├── 4.2 执行逆向三阶分析（怎么做→为什么→是什么）
    ├── 4.3 输出四层基础设施分析
    ├── 4.4 生成模块级文档（01-{模块A}-三阶解构.md）
    ├── 4.5 检查该模块是否还有 ✅需深入 的子模块
    │   └── 如有，递归执行第4步（序号递增：02-, 03-...）
    └── 4.6 当前模块所有 ✅ 分析完毕，返回上一级
           ↓
    所有模块递归分析完毕
           ↓
    最终输出：完整的项目文档集合
```

## 失败模式编码

> **设计依据**：借鉴 Darwin Skill dim3（失败模式编码）+ dim4（检查点显性标记）。只写正向流程的Skill是不完整的。

| 步骤 | 失败场景 | 处理动作 |
|------|---------|---------|
| 第1步 怎么做 | GitHub仓库无法访问 | 尝试本地克隆；若仍失败则询问用户是否有本地副本 |
| 第1步 怎么做 | 项目无README/文档 | 从源码结构推断，标注"无文档，推断可能不准" |
| 第1步 怎么做 | 代码量过大无法全读 | 优先读入口文件+核心模块，其余标注"未深入" |
| 第2步 为什么 | 设计意图无法从代码推导 | 标注"设计意图不明"，提供2-3种可能解释 |
| 第2步 为什么 | 技术选型原因不可知 | 标注"选型原因推测"，不强行确定 |
| 第3步 是什么 | 本质特征推导出多个矛盾结论 | 保留多个视角，标注"存在多种解读" |
| 第3步 是什么 | 系统边界模糊 | 标注"边界待确认"，列出可能的边界 |
| 第4步 递归深入 | 模块源码缺失/被混淆 | 标注"源码不可读"，跳过该模块 |
| 第4步 递归深入 | 递归超过3层仍未终止 | 强制终止，标注"深入不足" |

## 检查点

- 🔴 **CHECKPOINT · 第1步完成后**：确认项目结构和模块划分已识别，关键模块已标记✅
- 🔴 **CHECKPOINT · 第2步完成后**：确认设计推导已完成，无法推导的部分已标注
- 🔵 **AUTO-CHECK · 第3步完成后**：输出本质推导结果摘要，自动继续递归深入（用户可随时打断）
- 🔴 **CHECKPOINT · 每个模块递归分析后**：确认四层分析完整，无遗漏
