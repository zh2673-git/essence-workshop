# 法律分析 · 场景编排器

> **职责**：本场景只负责编排调用顺序（调用本目录 skills/ 下的原子技能），不直接实现认知分析。认知框架由 content-framework/law-application 定义，输出格式由 format 处理。

## 法律声明

> 本场景输出的是结构化法律分析草稿，**不构成法律意见，不替代执业律师**。报告中的法条引用、案例引用、解释结论均需通过官方法律数据库复核确认。重大决策前请咨询执业法律专业人员。

---

## 一、编排定位

本场景通过调度以下认知引擎原子技能完成端到端任务（原子技能位于 [./skills/](./skills/)）：

```
fact-trimming（事实裁剪）
  → legal-relationship-typing（法律关系定性）
  → norm-retrieval（规范检索）
  → case-retrieval（案例检索）
  → subsumption（涵摄）
  → evidence-discovery（证据发现）
  → legal-interpretation（法律解释）
  → conflict-resolution（冲突裁决）
  → defense-and-validity-review（抗辩与效力检视）
       ↓ 若用户已有证据 → evidence-evaluation（证据评估）
  → legal-consequence-calculation（法律后果量化）
  → fallback-application（兜底适用）
```

## 二、质量检查点（QC）

| 检查点 | 位置 | 检查内容 | 失败动作 |
|--------|------|----------|----------|
| QC1 | 事实裁剪后 | 要素是否完整、是否遗漏关键事实 | 回溯至用户，要求补充 |
| QC2 | 法律关系定性后 | 领域判断是否明确、行政行为三要件是否检视 | 回溯至事实裁剪 |
| QC3 | 规范检索后 | 候选规范是否覆盖争议焦点、来源是否标注 | 回溯至法律关系定性 |
| QC4 | 涵摄后 | 要件是否满足、中项是否明确、逻辑是否有效 | 回溯至规范检索或法律解释 |
| QC5 | 证据发现后 | 取证方案是否覆盖全部待证要件 | 回溯至 subsumption 补全 |
| QC6 | 冲突裁决后 | 解决方案是否一致、是否排除合理方案 | 回溯至涵摄 |
| QC7 | 抗辩检视后 | 抗辩是否影响结论、证据是否审查 | 回溯至涵摄或证据评估 |
| QC8 | 兜底适用后 | 是否确已穷尽具体规则 | 回溯至规范检索 |
| QC9 | 终稿前 | 报告完整性、置信度标注、免责声明 | 补充缺失内容 |

## 三、场景路由

| 场景 | 特征 | 阶段特殊处理 |
|------|------|--------------|
| 民事纠纷 | 平等主体、权利义务争议 | 启用九阶请求权检视 |
| 刑事案件 | 涉嫌犯罪、国家追诉 | 启用罪名检索、量刑计算 |
| 行政案件 | 行政行为三要件满足 | 自动检索行政规范性文件 |
| 宪法基本权利 | 涉及基本权利限制 | 启用比例原则、法律保留审查 |
| 复合型案件 | 刑民/行民交叉 | 并行多领域分析 |

## 四、标准调用链

```
输入：用户事件描述
  │
  ▼
S1  fact-trimming（事实裁剪）
  ↓ QC1
  ├─ 要素完整 → 继续
  └─ 要素缺失 → 要求用户补充
  ▼
S2  legal-relationship-typing（法律关系定性）
  ↓ QC2
  ├─ 领域明确 → 继续
  └─ 领域不明/竞合 → 标注并行路径
  ▼
S3  norm-retrieval（规范检索）
  ↓ QC3
  ├─ 规范覆盖完整 → 继续
  └─ 规范缺失 → 回溯或提示
  ▼
S4  case-retrieval（案例检索）
  ▼
S5  subsumption（涵摄）
  ↓ QC4
  ├─ 逻辑有效 → 继续
  └─ 逻辑存疑 / 要件缺失 → 回溯至 S3 或要求补充事实
  ▼
S5b evidence-discovery（证据发现）
  ↓ QC5
  ├─ 取证方案覆盖完整 → 继续
  └─ 覆盖不完整 → 回溯至 S5 补全要件
  ▼
S6a legal-interpretation（法律解释）
  ↓ 若无非明确规范则输出"本阶段无需法律解释"
  ▼
S6b conflict-resolution（冲突裁决）
  ↓ QC6
  ├─ 存在规范冲突 → 输出裁决结论
  └─ 无冲突 → 输出"本阶段无需冲突裁决"
  ▼
S7  defense-and-validity-review（抗辩与效力检视）
  │   ├─ 有证据 → evidence-evaluation（对照 S5b 取证方案评估）
  │   └─ 无证据 / 证据不足 → 基于 S5b 取证方案给出取证建议
  ↓ QC7
  ├─ 抗辩成立 → 调整结论，必要时回溯
  └─ 抗辩不成立 → 继续
  ▼
S8  legal-consequence-calculation（法律后果量化）
  ↓ 若无法量化则输出"本阶段无法量化，仅作定性分析"
  ▼
S9  fallback-application（兜底适用）
  ↓ QC8
  ├─ 已找到具体规则 → 输出"本阶段无需兜底适用"
  └─ 穷尽具体规则 → 输出兜底方案
  ▼
输出：结构化法条适用分析报告（格式由 format/markdown 处理）
```

## 五、条件分支与回溯机制

### 5.1 信息不足分支

```
if 用户描述信息不足以完成事实裁剪:
    输出已提取要素 + 关键待补充问题
    暂停流程，等待用户补充
    补充后从 S1 重新开始
```

### 5.2 行政行为分支

```
if legal-relationship-typing 判断为行政行为:
    norm-retrieval 自动进入第二层行政规范性文件检索
else:
    仅在用户明确要求时检索行政规范性文件
```

### 5.3 法律解释分支

```
默认调用 legal-interpretation
if 存在需要解释的非明确规范:
    输出解释对象、方法、结论
else:
    输出"本阶段无需法律解释"
```

### 5.4 冲突裁决分支

```
默认调用 conflict-resolution
if 存在多个可适用规范:
    输出冲突类型、解决规则、结论
else:
    输出"本阶段无需冲突裁决"
```

### 5.5 抗辩成立分支

```
if defense-and-validity-review 发现抗辩成立:
    调整或推翻先前结论
    若涉及新的事实，回溯至 fact-trimming
```

### 5.6 兜底适用分支

```
默认调用 fallback-application
if 无具体规则可涵摄:
    输出穷尽具体规则说明与兜底方案
else:
    输出"本阶段无需兜底适用"
```

## 八、失败模式与降级

| 失败场景 | 处理动作 |
|----------|----------|
| 用户输入完全无法理解 | 输出澄清问题，要求重新描述 |
| 信息严重不足 | 输出待补充清单，暂停流程 |
| MCP 与联网搜索均不可用 | 使用模型内部知识给出框架，全部法条标注 `[待校验]` |
| 全流程后仍无法得出结论 | 明确标注"无法形成确定性结论"，建议咨询执业律师 |
| 某一要件无对应取证材料 | 标注"证据缺位"，建议举证责任倒置、法律推定或申请法院调取 |
| 行政机关拒不提供材料 | 给出政府信息公开、行政复议、行政诉讼三条救济路径 |

## 九、各原子技能验证规则（随原子技能从 content-framework 迁入工作流层）

> 以下验证规则表原位于各原子技能的 SKILL.md 中，作为 QC 检查点迁入工作流层。

### fact-trimming（事实裁剪）

| 检查项 | 通过标准 |
|--------|----------|
| 最小单元化 | 每个事实单元包含主体、行为、对象、结果等核心要素 |
| 类型完整 | 至少覆盖主体、行为、时间、结果四类事实 |
| 属性净化 | 主观判断和法律评价已还原为客观事实或明确标注 |
| 待确认标注 | 缺失或矛盾事实已明确列出 |
| 法律语言中性 | 转化后的表述不含情绪词和评价词 |

### legal-relationship-typing（法律关系定性）

| 检查项 | 通过标准 |
|--------|----------|
| 主体关系明确 | 已识别各方主体及其相互关系 |
| 行为性质匹配 | 行为性质判断与事实要素一致 |
| 行政行为三要件完整 | 若涉及行政机关，已逐条检视三要件 |
| 救济路径可行 | 救济路径与主领域匹配 |
| 竞合已标注 | 存在竞合时明确列出并行分析路径 |

### norm-retrieval（规范检索）

| 检查项 | 通过标准 |
|--------|----------|
| 检索顺位正确 | 已按领域顺位从高位阶到低位阶检索 |
| 来源标注完整 | 每条规范均标注来源与检索时间 |
| 效力状态明确 | 已标注 VALID / AMENDED / REPEALED / UNCERTAIN 等 |
| 行政规范性文件条件正确 | 仅在行政行为三要件满足时自动检索 |
| 民事请求权无遗漏 | 已按九阶顺序检视 |

### subsumption（涵摄）

| 检查项 | 通过标准 |
|--------|----------|
| 要件完整 | 已分解规范的全部构成要件 |
| 事实对应 | 每个要件都有对应事实 |
| 中项明确 | 已识别连接规范与事实的中项 |
| 逻辑有效 | 三段论形式有效，无常见谬误 |
| 不确定性标注 | 存疑要件已明确标注 |

### conflict-resolution（冲突裁决）

| 检查项 | 通过标准 |
|--------|----------|
| 冲突分类正确 | 已正确识别冲突类型 |
| 规则适用正确 | 已按正确顺位适用冲突解决规则 |
| 排除理由充分 | 已说明为何排除其他方案 |
| 一致性 | 解决方案不自相矛盾 |
| 争点排序合理 | 排序符合程序/前提/主要优先规则 |

### legal-interpretation（法律解释）

| 检查项 | 通过标准 |
|--------|----------|
| 方法顺位 | 已按文义→体系→目的→历史顺序展开 |
| 权威支撑 | 已检索有权解释、立法说明、案例、学者通说 |
| 来源标注 | 所有支撑来源均已标注 |
| 结论限定 | 已说明解释结论的开放性与局限 |

### defense-and-validity-review（抗辩与效力检视）

| 检查项 | 通过标准 |
|--------|----------|
| 抗辩完整 | 已识别对方全部主要抗辩 |
| 证据三性 | 已审查关键证据的真实性、合法性、关联性 |
| 抗辩归类正确 | 已正确归入权利障碍/消灭/阻止 |
| 效力判断有据 | 效力判断有明确的法律依据 |
| 影响说明 | 已说明抗辩对原结论的影响 |

### fallback-application（兜底适用）

| 检查项 | 通过标准 |
|--------|----------|
| 穷尽具体规则 | 已列出并排除相关具体规则 |
| 漏洞论证充分 | 已说明为何存在漏洞 |
| 选择依据 | 已说明选择该兜底规范或原则的理由 |
| 限制条件 | 已列明适用限制 |
| 低置信度标注 | 兜底结论已明确标注低置信度 |

### case-retrieval（案例检索）

| 检查项 | 通过标准 |
|--------|----------|
| 来源标注 | 每个案例均标注案号、法院、来源 URL |
| 层级区分 | 已区分指导性案例、公报案例、普通裁判 |
| 相似度评估 | 已评估与本案的相似度 |
| 规则提炼 | 已提炼可复用的裁判规则 |

### evidence-discovery（证据发现）

| 检查项 | 通过标准 |
|--------|----------|
| 要件覆盖完整 | 每个待证要件都有对应证据材料 |
| 获取主体明确 | 每条材料都标注保管/制作主体 |
| 渠道可执行 | 获取方式具体，非空泛建议 |
| 证明力分级 | 已按强/中/弱分级并说明理由 |
| 替代方案 | 对难以取得的主证据提供替代路径 |
| 风险提示 | 已标注时效、取得、证明力、程序四类风险 |

### evidence-evaluation（证据评估）

| 检查项 | 通过标准 |
|--------|----------|
| 证据分类正确 | 已按法定证据种类分类 |
| 三性审查完整 | 每个证据均审查真实性、合法性、关联性 |
| 证明力评估合理 | 已结合证据种类、来源、利害关系评估 |
| 证明标准匹配 | 已按民事/刑事/行政标准判断 |

### legal-consequence-calculation（法律后果量化）

| 检查项 | 通过标准 |
|--------|----------|
| 公式正确 | 已使用法定或通行计算方式 |
| 调整因素完整 | 已列明法定与酌定因素 |
| 地区标准说明 | 已说明适用地区及标准差异 |
| 区间合理 | 已给出合理区间而非单一数值 |
| 风险提示 | 已说明计算结果仅供参考 |

## 十、法条适用框架运行流程与数据流转（从 content-framework/law-application/references/essence 迁入）

> 以下内容原位于 essence 思考归档中，属于执行编排，迁入工作流层。

### 框架关系与运行流程（迁自 01-what-essence.md）

```
[事实裁剪框架]
       │
       ↓
[法律关系定性框架]
       │
       ↓
[规范检索框架] ──→ [涵摄框架]
       │                   │
       ↓                   ↓
[冲突裁决框架]      [抗辩与效力检视框架]
       │                   │
       ↓                   ↓
[兜底适用框架] ←──→ [法律解释框架]
```

**关键顺序**：
1. 先裁剪事实，再定性法律关系
2. 先定性法律关系，再按领域检索规范
3. 先检索规范，再涵摄构成要件
4. 先涵摄成立，再检查抗辩/效力
5. 规范冲突时启动冲突裁决
6. 穷尽规则时启用兜底适用或法律解释

### 数据流转与场景路由分发（迁自 03-how-system.md）

```
用户输入（事件/问题）
  → 路由分发
    → 场景A：事实裁剪 → 法律关系定性 → 规范检索（先正式法源；行政行为再补规范性文件） → 涵摄 → 抗辩与效力检视 → 输出结论
    → 场景B：法律关系定性 → 涵摄 → 抗辩与效力检视 → 输出结论
    → 场景C：位阶判断 → 冲突裁决 → 输出适用法条
    → 场景D：具体规则穷尽 → 兜底/漏洞填补 → 输出结论
    → 场景E：文义解释 → 体系解释 → 目的解释 → 输出解释结论
    → 默认：事实裁剪 → 法律关系定性 → 规范检索（先正式法源；行政行为再补规范性文件） → 涵摄 → 输出结论
```

### A.1 调用链补充说明

`skills/orchestrator` 是端到端分析的默认入口，负责：接收用户输入、按标准流程调度原子 skill、设置质量检查点（QC）、处理条件分支与回溯、输出完整的法条适用分析报告。

标准调用链（与上文「四、标准调用链」一致）：

```
fact-trimming
  → legal-relationship-typing
  → norm-retrieval
  → [case-retrieval]
  → subsumption
  → [legal-interpretation]
  → [conflict-resolution]
  → defense-and-validity-review
  → [legal-consequence-calculation]
  → [fallback-application]
```

### A.2 检索策略执行规则：MCP 优先，联网搜索兜底

法条与案例检索采用**分层降级策略**：

| 层级 | 方式 | 说明 |
|------|------|------|
| 第一层 | 已配置 MCP 工具 | 北大法宝 MCP、秘塔搜索 MCP、其他法律法规 MCP |
| 第二层 | 模型/Agent 自身联网搜索 | 国家法律法规数据库、裁判文书网、人民法院案例库等 |
| 第三层 | 模型内部知识 | 仅作框架性提示，所有引用标注 `[待校验]` |

**校验要求**：

- 所有法条、案例引用必须标注来源、检索时间、效力状态；
- 优先通过至少两个独立来源交叉验证；
- 无法核验的引用标注 `[待校验]` 或 `[效力存疑]`。

### A.3 使用方式

| 方式 | 入口 | 适用场景 |
|------|------|---------|
| 方式一：总入口（推荐） | 激活本编排器 `workflow/law-analysis/SKILL.md`，由其路由到默认入口编排器 | 常规端到端分析 |
| 方式二：复合编排器 | 直接激活本编排器，输入具体事件描述 | 端到端分析 |
