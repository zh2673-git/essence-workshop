name: orchestrator
description: |
  当用户提交一个具体法律问题或案件，需要端到端的法条适用分析时触发本技能。
  本技能是 law-application.skill 的复合编排器（位于 skills/orchestrator/），负责按标准流程调度原子技能，设置质量检查点（QC），处理条件分支与回溯。
  典型触发场景：
  - 用户描述具体事件，希望获得完整法律分析；
  - 用户希望了解某一行为在法律上的后果；
  - 用户需要判断权利义务、救济路径或法律风险。
  编排器默认执行完整流程（事实裁剪 → 法律关系定性 → 规范检索 → 案例检索 → 涵摄 → 证据发现 → 法律解释 → 冲突裁决 → 抗辩检视 → 法律后果量化 → 兜底适用），每个阶段都会呈现在最终报告中；若某阶段确实不适用，则标注"本阶段无需…"。
  本技能不直接输出最终法律意见，而是输出结构化分析报告，供执业法律专业人员审阅。
  边界：不能替代执业律师；涉及重大财产、人身自由或复杂程序事项时，必须建议咨询专业人士。

# 法条适用编排器 · 复合技能

## 法律声明

> 本技能输出的是结构化法律分析草稿，**不构成法律意见，不替代执业律师**。报告中的法条引用、案例引用、解释结论均需通过官方法律数据库复核确认。重大决策前请咨询执业法律专业人员。

---

## 一、核心概念

### 1.1 编排器定位

编排器本身不引入新的法律分析能力，它通过调度以下原子技能完成端到端任务：

```
fact-trimming（事实裁剪）
  → legal-relationship-typing（法律关系定性）
  → norm-retrieval（规范检索）
  → case-retrieval（案例检索）
  → subsumption（涵摄）
  → evidence-discovery（证据发现）              ← 新增：从要件推导取证方案
  → legal-interpretation（法律解释）
  → conflict-resolution（冲突裁决）
  → defense-and-validity-review（抗辩与效力检视）
       ↓ 若用户已有证据 → evidence-evaluation（证据评估）
  → legal-consequence-calculation（法律后果量化）
  → fallback-application（兜底适用）
```

### 1.2 质量检查点（QC）

| 检查点 | 位置 | 检查内容 | 失败动作 |
|--------|------|----------|----------|
| QC1 | 事实裁剪后 | 要素是否完整、是否遗漏关键事实 | 回溯至用户，要求补充 |
| QC2 | 法律关系定性后 | 领域判断是否明确、行政行为三要件是否检视 | 回溯至事实裁剪 |
| QC3 | 规范检索后 | 候选规范是否覆盖争议焦点、来源是否标注 | 回溯至法律关系定性 |
| QC4 | 涵摄后 | 要件是否满足、中项是否明确、逻辑是否有效 | 回溯至规范检索或法律解释 |
| QC5 | 证据发现后 | 取证方案是否覆盖全部待证要件 | 回溯至 subsumption 补全 |
| QC6 | 冲突裁决后 | 解决方案是否一致、是否排除合理方案 | 回溯至涵摄 |
| QC7 | 抗辩检视后 | 抗辩是否影响结论、证据是否审查 | 回溯至涵摄或证据评估 |
| QC8 | 兜底适用后 | 是否确已穷尽具体规则 | 回溯至规范检索 |
| QC9 | 终稿前 | 报告完整性、置信度标注、免责声明 | 补充缺失内容 |

### 1.3 场景路由

| 场景 | 特征 | 阶段特殊处理 |
|------|------|--------------|
| 民事纠纷 | 平等主体、权利义务争议 | 启用九阶请求权检视 |
| 刑事案件 | 涉嫌犯罪、国家追诉 | 启用罪名检索、量刑计算 |
| 行政案件 | 行政行为三要件满足 | 自动检索行政规范性文件 |
| 宪法基本权利 | 涉及基本权利限制 | 启用比例原则、法律保留审查 |
| 复合型案件 | 刑民/行民交叉 | 并行多领域分析 |

## 二、标准调用链

```
输入：用户事件描述
  │
  ▼
S1  fact-trimming（事实裁剪）
  ↓ QC1
  ├─ 要素完整 → 继续
  └─ 要素缺失 → 要求用户补充
  ▼
S2  legal-relationship-typing（法律关系定性）
  ↓ QC2
  ├─ 领域明确 → 继续
  └─ 领域不明/竞合 → 标注并行路径
  ▼
S3  norm-retrieval（规范检索）
  │   ├─ 优先 MCP（北大法宝、秘塔等）
  │   ├─ 其次模型/Agent 联网搜索
  │   └─ 最后模型内部知识，标注 [待校验]
  ↓ QC3
  ├─ 规范覆盖完整 → 继续
  └─ 规范缺失 → 回溯或提示
  ▼
S4  case-retrieval（案例检索）
  ▼
S5  subsumption（涵摄）
  ↓ QC4
  ├─ 逻辑有效 → 继续
  └─ 逻辑存疑 / 要件缺失 → 回溯至 S3 或要求补充事实
  ▼
S5b evidence-discovery（证据发现）
  ↓ QC5
  ├─ 取证方案覆盖完整 → 继续
  └─ 覆盖不完整 → 回溯至 S5 补全要件
  ▼
S6a legal-interpretation（法律解释）
  ↓ 若无非明确规范则输出"本阶段无需法律解释"
  ▼
S6b conflict-resolution（冲突裁决）
  ↓ QC6
  ├─ 存在规范冲突 → 输出裁决结论
  └─ 无冲突 → 输出"本阶段无需冲突裁决"
  ▼
S7  defense-and-validity-review（抗辩与效力检视）
  │   ├─ 有证据 → evidence-evaluation（对照 S5b 取证方案评估）
  │   └─ 无证据 / 证据不足 → 基于 S5b 取证方案给出取证建议
  ↓ QC7
  ├─ 抗辩成立 → 调整结论，必要时回溯
  └─ 抗辩不成立 → 继续
  ▼
S8  legal-consequence-calculation（法律后果量化）
  ↓ 若无法量化则输出"本阶段无法量化，仅作定性分析"
  ▼
S9  fallback-application（兜底适用）
  ↓ QC8
  ├─ 已找到具体规则 → 输出"本阶段无需兜底适用"
  └─ 穷尽具体规则 → 输出兜底方案
  ▼
输出：结构化法条适用分析报告
```

## 三、条件分支与回溯机制

### 3.1 信息不足分支

```
if 用户描述信息不足以完成事实裁剪:
    输出已提取要素 + 关键待补充问题
    暂停流程，等待用户补充
    补充后从 S1 重新开始
```

### 3.2 行政行为分支

```
if legal-relationship-typing 判断为行政行为:
    norm-retrieval 自动进入第二层行政规范性文件检索
else:
    仅在用户明确要求时检索行政规范性文件
```

### 3.3 法律解释分支

```
默认调用 legal-interpretation
if 存在需要解释的非明确规范:
    输出解释对象、方法、结论
else:
    输出"本阶段无需法律解释"
```

### 3.4 冲突裁决分支

```
默认调用 conflict-resolution
if 存在多个可适用规范:
    输出冲突类型、解决规则、结论
else:
    输出"本阶段无需冲突裁决"
```

### 3.5 抗辩成立分支

```
if defense-and-validity-review 发现抗辩成立:
    调整或推翻先前结论
    若涉及新的事实，回溯至 fact-trimming
```

### 3.6 兜底适用分支

```
默认调用 fallback-application
if 无具体规则可涵摄:
    输出穷尽具体规则说明与兜底方案
else:
    输出"本阶段无需兜底适用"
```

## 五、示例

**输入**："我借给朋友 5 万元，约定 1 个月还，现在 3 个月没还，他把我微信删了，我能告他吗？"

**输出**（节选）：

```markdown
# 法条适用分析报告

## 一、案件摘要
出借人向借款人出借 5 万元，约定 1 个月归还，逾期 3 个月未还，借款人删除出借人微信。

## 三、法律关系定性
- 主领域：民事
- 子类型：借款合同纠纷
- 是否行政行为：否
- 救济路径：民事诉讼

## 四、规范检索
- 《民法典》第675条（返还借款）✅ VALID
- 《民法典》第676条（逾期利息）✅ VALID

## 六、涵摄分析
```
P: 借款人应按约定期限返还借款（《民法典》第675条）
F: 本案借款人约定1个月返还，已逾期3个月未还
∴ C: 出借人有权请求借款人返还借款
```

## 九、抗辩与效力检视
- 借款人可能主张：诉讼时效已过
- 审查结论：出借人有催告记录，时效中断，抗辩不成立

## 十二、综合结论
- **核心结论**：出借人有权向人民法院起诉，请求借款人返还借款本金及逾期利息。
- **置信度**：高
- **建议行动**：准备借条/转账记录/催告记录，向有管辖权的人民法院起诉。
```

## 六、验证规则

| 检查项 | 通过标准 |
|--------|----------|
| 流程完整 | 已按标准调用链执行 |
| QC 通过 | 各检查点均通过或已标注回溯 |
| 来源标注 | 法条、案例引用均已标注来源 |
| 置信度标注 | 不确定结论已标注置信度 |
| 免责声明 | 终稿包含免责声明 |

## 七、失败模式与降级

| 失败场景 | 处理动作 |
|----------|----------|
| 用户输入完全无法理解 | 输出澄清问题，要求重新描述 |
| 信息严重不足 | 输出待补充清单，暂停流程 |
| MCP 与联网搜索均不可用 | 使用模型内部知识给出框架，全部法条标注 `[待校验]` |
| 全流程后仍无法得出结论 | 明确标注"无法形成确定性结论"，建议咨询执业律师 |
| 某一要件无对应取证材料 | 标注"证据缺位"，建议举证责任倒置、法律推定或申请法院调取 |
| 行政机关拒不提供材料 | 给出政府信息公开、行政复议、行政诉讼三条救济路径 |
