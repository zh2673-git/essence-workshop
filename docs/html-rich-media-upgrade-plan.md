# 本质工坊 HTML 富媒体改造方案

> 基于「大模型时代的 HTML 复兴」思想与 TeachAny 项目实践，推动本质工坊从「文本交互」向「富媒体交付」进化。

---

## 一、改造背景与动机

### 1.1 核心矛盾

当前本质工坊的输出链路存在一个根本矛盾：**认知系统是丰富的、多维的，但交付方式是扁平的、文本的。**

| 维度 | 本质工坊现有能力 | 交付方式 |
|------|----------------|---------|
| 知识探索 | 结构化知识体系（概念→联系→细节） | Markdown 文本笔记 |
| 人物蒸馏 | 6个心智模型 + 7条启发式 + 场景路由 | Markdown Skill 文件 |
| 项目开发 | 设计文档 + 可运行代码 | Markdown + 代码文件 |
| 内容输出（公众号） | 图文排版 + SVG配图 + GIF动图 | 微信 HTML（内联样式） |
| 内容输出（视频号） | Canvas卡片翻页 + TTS + FFmpeg | MP4 视频 |

公众号输出已经是 HTML，但它是「微信受限 HTML」——无交互、无动画、无动态数据。视频号输出已经是富媒体，但它是「录制式视频」——不可交互、不可复用、修改成本高。

更深层的问题是：**公众号配图和视频号画面是两套独立生成的，同一张知识图谱在公众号里是 SVG→PNG 静态图，在视频号里是 Canvas 重新绘制——元素没有复用，修改要改两处。**

### 1.2 文章核心洞察

微信文章《当大模型开始输出 HTML 而不是 Markdown》提出了一个关键判断：

> **Markdown 解决"快速传递"的问题，HTML 解决"高效消费"的问题。两者不是取代关系，而是场景分化。**

文章的六维信息容器理论：

| 维度 | Markdown | HTML |
|------|----------|------|
| 内容 | ✅ | ✅ |
| 结构 | ✅（有限） | ✅（完整） |
| 布局 | ❌ | ✅ |
| 样式 | ❌ | ✅ |
| 交互 | ❌ | ✅ |
| 保真度 | ❌ | ✅ |

**判断标准**：如果读者只需要"读"，Markdown 足够；如果读者需要"用"（交互、调整、探索），HTML 才是正确选择。

文章还给出了图形技术选型的决策框架——SVG/Canvas/WebGL/CSS 各有适用场景，选择标准是渲染模型和元素规模，而非主观偏好。

### 1.3 TeachAny 的验证

TeachAny 项目用实践验证了这一判断：

- **模板优先**：所有课件从 `course-skeleton-v2.html` 模板出发，不手写平台接线
- **标准模块**：AI 学伴、TTS、知识图谱、导师卡片、Section Hints 五件套
- **互动闭环**：问题锚点 → 互动尝试 → 即时反馈 → 总结迁移
- **发布体系**：Registry 注册 → Gallery 展示 → CDN 分发
- **质量门禁**：`validate-courseware.cjs` 质检，19 项基线不可省略

TeachAny 证明了：**HTML 不仅是展示格式，更是学习体验的载体。**

## 二、核心架构思想：元素层 → 管线层 → 平台层

### 2.1 关键认知转变

原方案的问题是：公众号管线和视频号管线各自独立生成富媒体元素，同一内容在两条管线里被重复创造、格式不同、无法复用。

**正确的架构是三层分离**：

```
┌─────────────────────────────────────────────────────────────┐
│                    平台层（Platform）                         │
│   微信公众号 │ 微信视频号 │ HTML交互页 │ 未来平台X           │
│   各平台有各自的约束和格式要求                                │
└──────────────────────┬──────────────────────────────────────┘
                       │ 消费
┌──────────────────────┴──────────────────────────────────────┐
│                    管线层（Pipeline）                         │
│   公众号管线 │ 视频号管线 │ HTML交互管线 │ 演示管线 │ ...    │
│   每条管线 = 从元素层取原子资产 → 按平台约束组装 → 输出成品  │
└──────────────────────┬──────────────────────────────────────┘
                       │ 读取
┌──────────────────────┴──────────────────────────────────────┐
│                    元素层（Element）                          │
│                                                              │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│   │ 文本元素 │ │ 图形元素 │ │ 动画元素 │ │ 音频元素 │      │
│   │ 标题/段落│ │ SVG图谱  │ │ SVG动画  │ │ TTS旁白  │      │
│   │ 列表/表格│ │ SVG流程图│ │ Canvas帧 │ │ BGM      │      │
│   │ 代码块   │ │ SVG架构图│ │ CSS动画  │ │ 音效     │      │
│   └──────────┘ └──────────┘ └──────────┘ └──────────┘      │
│                                                              │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐                    │
│   │ 视频元素 │ │ 交互元素 │ │ 数据元素 │                    │
│   │ 视频片段 │ │ 拖拽组件 │ │ JSON数据 │                    │
│   │ 录屏片段 │ │ 折叠面板 │ │ 配置参数 │                    │
│   │ 动画序列 │ │ 路由匹配 │ │ 状态对象 │                    │
│   └──────────┘ └──────────┘ └──────────┘                    │
│                                                              │
│   元素是原子资产：自包含、可复用、跨管线共享                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 与现有设计原则的呼应

这个三层架构与本质工坊已有的设计原则完全一致：

| 现有原则 | 三层架构中的体现 |
|---------|----------------|
| **单一状态源** | 元素层就是状态源，所有管线读取同一组元素 |
| **四层基础设施** | 元素层 = 数据规矩+存储，管线层 = 数据流转，平台层 = 接口层 |
| **类-属性-方法-路由** | 元素 = 属性，管线 = 方法，平台 = 路由分发 |

### 2.3 元素复用示例

以「知识图谱」为例，看同一元素如何在不同管线中复用：

```
元素层：knowledge-graph.svg（SVG 知识图谱，含节点+边+交互事件）

公众号管线读取：
  SVG → 转PNG → 内联到微信HTML → 静态配图

视频号管线读取：
  SVG → Canvas渲染 → Playwright录制 → 合入视频帧

HTML交互管线读取：
  SVG → 直接嵌入HTML → 保留交互（拖拽/点击/悬停）

演示管线读取：
  SVG → 嵌入Reveal.js幻灯片 → 翻页时展示

未来平台X管线读取：
  SVG → 按平台X要求转换 → 输出
```

**核心收益**：元素只生成一次，修改只改一处，所有管线自动获得更新。

### 2.4 一句话目标

**将本质工坊从「每条管线独立生成内容」升级为「元素层统一生成原子资产 → 管线层按平台约束组装 → 平台层交付」。**

## 三、元素层设计

### 3.1 元素分类体系

| 类别 | 原子元素 | 技术载体 | 生成方式 |
|------|---------|---------|---------|
| **文本元素** | 标题、段落、列表、表格、代码块、引用 | Markdown / HTML | AI 直接生成 |
| **图形元素** | 知识图谱、流程图、架构图、对比图、心智模型图 | SVG | AI 生成 SVG 代码 |
| **动画元素** | 状态转换动画、路径动画、手绘效果、粒子效果 | SVG 动画 / Canvas 帧 / CSS 动画 | AI 生成代码 |
| **音频元素** | TTS 旁白、BGM、音效 | MP3 | Edge TTS / 本地文件 |
| **视频元素** | 视频片段、录屏、动画序列 | WebM / MP4 | Playwright 录制 / FFmpeg |
| **交互元素** | 拖拽组件、折叠面板、深度切换、场景路由 | JS 模块 | 标准模块库 |
| **数据元素** | 知识结构JSON、配置参数、状态对象 | JSON | AI 生成 |

### 3.2 元素的原子性规则

1. **自包含**：每个元素可独立存在，不依赖其他元素
2. **可寻址**：每个元素有唯一标识（文件名或ID），管线可按需读取
3. **可组合**：元素之间可以组合，但组合在管线层完成，不在元素层
4. **可转换**：同一元素可被不同管线转换为不同格式（SVG→PNG、SVG→Canvas帧）

### 3.3 元素存储结构

```
output/
└── {project-name}/
    ├── elements/                    # 元素层：原子资产
    │   ├── text/                    # 文本元素
    │   │   ├── 01-intro.md
    │   │   ├── 02-core-concept.md
    │   │   └── 03-summary.md
    │   ├── graphics/                # 图形元素
    │   │   ├── knowledge-graph.svg  # 知识图谱（SVG，可交互）
    │   │   ├── flow-chart.svg       # 流程图
    │   │   └── mental-model.svg     # 心智模型图
    │   ├── animations/              # 动画元素
    │   │   ├── state-transition.svg # SVG 动画
    │   │   └── particle-bg.html     # Canvas 粒子背景
    │   ├── audio/                   # 音频元素
    │   │   ├── tts-s01.mp3
    │   │   └── bgm.mp3
    │   ├── data/                    # 数据元素
    │   │   ├── structure.json       # 知识结构
    │   │   └── brand-spec.json      # 品牌规格
    │   └── manifest.json            # 元素清单（注册表）
    ├── wechat/                      # 公众号管线输出
    ├── video/                       # 视频号管线输出
    ├── html/                        # HTML交互管线输出
    └── slides/                      # 演示管线输出
```

## 四、管线层设计

### 4.1 管线总览

| 管线 | 输入（从元素层读取） | 输出 | 核心转换 |
|------|---------------------|------|---------|
| **公众号管线** | 文本+图形+动画 | 微信受限HTML | SVG→PNG内联，动画→GIF，文本→微信排版 |
| **视频号管线** | 文本+图形+动画+音频 | MP4视频 | SVG→Canvas帧，文本→卡片，音频→TTS+BGM |
| **HTML交互管线** | 文本+图形+动画+交互 | 完整HTML | SVG直接嵌入，交互模块接入，保留全部能力 |
| **演示管线** | 文本+图形+动画 | Reveal.js HTML | 文本→幻灯片section，图形嵌入幻灯片 |
| **PPT管线** | 文本+图形+动画 | .pptx文件 | SVG→PNG，文本→幻灯片，动画→关键帧截图 |
| **（未来扩展）** | ... | ... | 按新平台约束编写新管线 |

### 4.2 管线与元素的关系

```
                    ┌─────────────────────────────────────┐
                    │            元素层                     │
                    │  文本  图形  动画  音频  交互  数据   │
                    └──┬─────┬─────┬─────┬─────┬─────┬───┘
                       │     │     │     │     │     │
        ┌──────────────┼─────┼─────┼─────┼─────┤     │
        │ 公众号管线    │     │     │     │     │     │
        │ 读取:文本+图形+动画   │     │     │     │
        │ 转换:SVG→PNG,GIF     │     │     │
        │ 输出:微信HTML        │     │     │
        └──────────────┼─────┼─────┼─────┼─────┤     │
        │ 视频号管线    │     │     │     │     │     │
        │ 读取:文本+图形+动画+音频  │     │     │
        │ 转换:SVG→Canvas,文本→卡片    │     │
        │ 输出:MP4             │     │     │
        └──────────────┼─────┼─────┼─────┼─────┤     │
        │ HTML交互管线  │     │     │     │     │     │
        │ 读取:文本+图形+动画+交互+数据     │     │
        │ 转换:直接嵌入，保留交互           │     │
        │ 输出:完整HTML        │     │     │
        └──────────────┼─────┼─────┼─────┼─────┤     │
        │ 演示管线      │     │     │     │     │     │     │
        │ 读取:文本+图形+动画   │     │     │     │
        │ 转换:文本→section,图形嵌入       │     │
        │ 输出:Reveal.js HTML  │     │     │
        └──────────────┼─────┼─────┼─────┼─────┤     │
        │ PPT管线       │     │     │     │     │     │
        │ 读取:文本+图形+动画   │     │     │     │
        │ 转换:SVG→PNG,文本→幻灯片        │     │
        │ 输出:.pptx           │     │     │
        └──────────────┼─────┼─────┼─────┼─────┼─────┤
                       │     │     │     │     │     │
                    ┌──┴─────┴─────┴─────┴─────┴─────┴───┐
                    │         未来管线X                      │
                    │  读取:按需组合元素                      │
                    │  转换:按平台X约束                       │
                    │  输出:平台X格式                         │
                    └──────────────────────────────────────┘
```

### 4.3 各管线详细设计

#### 4.3.1 公众号管线（改造现有）

**现有流程**：Markdown → 微信HTML（内联样式）+ SVG→PNG配图 + Canvas→GIF动图

**改造后**：从元素层读取 → 按微信约束组装

```
元素层读取：
  text/*.md → 文章正文
  graphics/*.svg → SVG配图
  animations/*.svg → SVG动画
  data/brand-spec.json → 品牌色卡

公众号约束适配：
  SVG → PNG（微信不支持SVG外链，需内联或转PNG）
  SVG动画 → GIF（微信不支持SVG动画）
  交互元素 → 降级为静态图（微信不支持JS交互）
  文本 → 微信排版（内联样式，禁用:::block）

输出：
  wechat/article.html（微信受限HTML）
  wechat/cover.png（封面）
```

**改造要点**：公众号管线不再独立生成 SVG 配图，而是从元素层读取已有的 SVG 元素，转换为 PNG/GIF 后嵌入。同一张知识图谱，在公众号里是静态 PNG，在 HTML 交互页里是可交互 SVG——元素是同一个，只是管线的转换方式不同。

#### 4.3.2 视频号管线（改造现有）

**现有流程**：文章 → 镜头拆分 → Canvas渲染+Playwright录制 → TTS+FFmpeg合并

**改造后**：从元素层读取 → 按视频约束组装

```
元素层读取：
  text/*.md → 旁白文本 → TTS
  graphics/*.svg → 视觉画面素材
  animations/* → 动画序列
  audio/tts-*.mp3 → 旁白音频
  audio/bgm.mp3 → 背景音乐
  data/structure.json → 镜头拆分依据

视频约束适配：
  SVG → Canvas渲染（Playwright录制）
  文本 → 卡片画面（标题卡/要点卡/金句卡等）
  音频 → TTS+BGM+Ducking混音

输出：
  video/output.mp4（视频号竖屏视频）
```

**改造要点**：视频号管线不再独立绘制知识图谱，而是从元素层读取 SVG，通过 Canvas 渲染后录制。修改知识图谱只需改元素层的 SVG，视频号管线重新渲染即可。

#### 4.3.3 HTML交互管线（新增核心）

**流程**：从元素层读取 → 直接嵌入HTML，保留全部能力

```
元素层读取：
  text/*.md → 文章正文
  graphics/*.svg → 直接嵌入（保留交互）
  animations/*.svg → 直接嵌入（保留动画）
  交互模块 → 接入标准模块
  data/structure.json → 知识结构数据
  data/brand-spec.json → 主题配置

HTML约束适配：
  无降级！SVG保留交互，动画保留播放，模块保留功能
  响应式布局适配手机/桌面
  主题切换（暗色/亮色/暖色/自然）

输出：
  html/index.html（完整交互HTML）
```

**这是唯一保留元素全部能力的管线**。公众号管线是「降级交付」，视频号管线是「录制交付」，HTML交互管线是「完整交付」。

#### 4.3.4 演示管线（新增）

**流程**：从元素层读取 → 组装为 Reveal.js 幻灯片

```
元素层读取：
  text/*.md → 拆分为幻灯片section
  graphics/*.svg → 嵌入对应幻灯片
  animations/* → 嵌入对应幻灯片

演示约束适配：
  H1 → 封面幻灯片
  H2 → 章节分隔幻灯片
  H3 → 内容幻灯片
  SVG配图 → 嵌入对应幻灯片
  代码块 → 代码高亮幻灯片

输出：
  slides/index.html（Reveal.js演示）
```

#### 4.3.5 PPT管线（新增）

**流程**：从元素层读取 → 组装为 .pptx 文件

```
元素层读取：
  text/*.md → 幻灯片文本内容
  graphics/*.svg → 幻灯片图片
  animations/* → 动画关键帧截图
  data/structure.json → 幻灯片顺序和层级

PPT约束适配：
  Markdown H1 → 封面幻灯片（标题+副标题）
  Markdown H2 → 章节分隔幻灯片
  Markdown H3 → 内容幻灯片
  SVG → PNG → PPT图片（SVG无法直接嵌入pptx）
  SVG动画 → 截取关键帧 → PPT图片序列或静态截图
  交互元素 → 降级为静态截图（PPT不支持JS交互）
  代码块 → PPT文本框（等宽字体）

输出：
  pptx/output.pptx（PowerPoint文件）
```

**技术方案**：使用 `python-pptx` 或 `PptxGenJS` 从元素层生成 .pptx 文件。

**与演示管线的关系**：

| 维度 | 演示管线（Reveal.js） | PPT管线（.pptx） |
|------|---------------------|-----------------|
| 交互能力 | 完整（SVG交互+JS模块） | 无（静态截图） |
| 格式 | HTML（浏览器打开） | .pptx（PowerPoint/WPS打开） |
| 版本管理 | Git友好（纯文本） | Git不友好（二进制） |
| AI生成 | 原生友好 | 需要库辅助 |
| 离线使用 | 需要浏览器 | 直接打开 |
| 企业模板 | 不支持 | 支持（可套用.potx模板） |
| 适用场景 | 技术分享、在线演示 | 企业汇报、客户交付、离线环境 |

**推荐策略**：优先使用演示管线（Reveal.js），仅在以下场景使用 PPT 管线：
- 客户/公司明确要求 .pptx 格式
- 需要套用企业 .potx 模板
- 离线环境无浏览器
- 需要他人编辑幻灯片内容

## 五、标准交互模块库

### 5.1 模块定位

标准模块是**交互元素的预制件**——它们不是管线，而是可以被 HTML交互管线直接使用的交互组件。公众号管线和视频号管线不使用这些模块（因为微信和视频不支持JS交互），但它们消费的图形元素（SVG）可以与模块共享同一套数据源。

```
元素层
  ├── 图形元素（SVG）← 公众号管线读取→转PNG
  │                    ← 视频号管线读取→Canvas渲染
  │                    ← HTML交互管线读取→嵌入+接入交互模块
  │
  └── 交互模块（JS）← 仅HTML交互管线使用
```

### 5.2 模块清单

| 模块 | 功能 | 消费的元素 | 适用场景 |
|------|------|-----------|---------|
| **坡度导航器** | 阶段1/2/3 渐进展开 | 文本元素 | 所有场景 |
| **三阶进度条** | 是什么→为什么→怎么做 进度指示 | 数据元素 | 所有场景 |
| **知识图谱浏览器** | 可拖拽/缩放/点击的知识图谱 | 图形元素(SVG) | 场景A/B |
| **心智模型卡片** | 可展开的心智模型详情卡 | 图形元素+文本元素 | 场景B |
| **场景路由器** | 输入场景→匹配启发式 | 数据元素 | 场景B |
| **品牌主题系统** | 暗色/亮色/暖色/自然 4主题 | 数据元素(brand-spec) | 所有HTML输出 |
| **根因追溯线** | 时间线式根因追溯可视化 | 图形元素+文本元素 | 场景B |

### 5.3 模块封装方式

每个模块自包含为单个 JS 文件，通过 `data-*` 属性配置：

```html
<div data-module="slope-navigator"
     data-stages='["点出概念","建立联系","详细解释"]'
     data-theme="dark">
</div>
<script src="modules/slope-navigator.js"></script>
```

**设计原则**：
- 单文件部署，无需构建工具
- `data-*` 配置对 AI 生成友好
- 通过 CustomEvent 与外部通信
- 支持主题切换（CSS 变量）
- 无全局污染

## 六、图形技术选型

基于文章提出的技术选型原则，为元素层的图形元素制定选型规则：

### 6.1 选型决策树

```
内容是否有时间维度？
├── 否（静态结构/关系/对比）
│   └── SVG（首选）
│       ├── 元素 < 5000 → SVG 直接用
│       └── 元素 > 5000 → SVG + 虚拟DOM优化（SSVG）
│
└── 是（有变化过程/状态转换/运动）
    ├── 简单动画（< 1000元素）
    │   └── SVG 动画（CSS/SMIL）
    └── 复杂动画（> 1000元素 / 高帧率）
        └── Canvas

需要3D空间？
└── WebGL（Three.js）— 预留，暂不实施
```

### 6.2 元素类型与技术映射

| 元素类型 | 技术载体 | 理由 | 管线适配 |
|---------|---------|------|---------|
| 知识图谱 | SVG | DOM可交互，<5000节点 | 公众号→PNG，视频号→Canvas渲染，HTML→直接嵌入 |
| 流程图 | SVG | 静态结构，精确布局 | 同上 |
| 心智模型图 | SVG | 节点少，需交互 | 同上 |
| 架构分层图 | SVG | 静态结构 | 同上 |
| 状态转换动画 | SVG动画 | 简单动画，<1000元素 | 公众号→GIF，视频号→Canvas录制，HTML→直接嵌入 |
| 粒子/高频动画 | Canvas | 性能要求 | 公众号→GIF，视频号→直接录制，HTML→Canvas嵌入 |
| 视频号卡片 | Canvas | 已有成熟管线 | 仅视频号管线使用 |

### 6.3 SVG 优先原则

**SVG 是元素层的首选图形格式**，原因：

1. **文本即图形**：AI 直接生成 SVG 代码，token 成本低
2. **矢量无损**：放大缩小不模糊，跨设备一致
3. **DOM可交互**：每个元素是独立DOM节点，支持事件绑定
4. **管线友好**：SVG 可被所有管线消费（→PNG / →Canvas帧 / →直接嵌入）
5. **动画支持**：CSS/SMIL 动画，简单场景不需要 Canvas

**Canvas 仅用于**：视频号卡片渲染（已有管线）、万级元素场景、高频粒子动画。

## 十一、关键设计决策

### 决策1：元素层是单一状态源

**决策**：元素层是所有管线的唯一数据源，管线不独立生成元素。

这与本质工坊已有的「单一状态源」原则完全一致——项目级有全局状态，模块级有内部状态。元素层就是项目级全局状态，管线就是读取状态的模块。

**例外**：管线内部的格式转换（SVG→PNG、SVG→Canvas帧）是管线自己的内部状态，不需要回写到元素层。

### 决策2：SVG是元素层的首选图形格式

**决策**：所有图形元素优先以 SVG 生成和存储。

理由：
- AI 直接生成 SVG 代码，token 成本低
- SVG 可被所有管线消费（→PNG / →Canvas帧 / →直接嵌入）
- SVG 是矢量，放大不模糊
- SVG 是 DOM，天然可交互
- SVG 是文本，Git 可 diff

Canvas 仅用于视频号卡片渲染和万级元素场景。

### 决策3：Markdown是文本元素的源格式

**决策**：文本元素以 Markdown 生成和存储，管线负责转换为各平台格式。

理由：
- AI 生成 Markdown 的 token 效率远高于 HTML
- Markdown 可读性好，适合版本控制
- 各管线按需转换：公众号→微信HTML，视频号→卡片文本，HTML交互→完整HTML

### 决策4：模板优先

**决策**：所有管线输出从模板出发，不手写平台接线。

借鉴 TeachAny 的模板优先原则：
- 模板包含：标准模块接入、主题系统、响应式布局、无障碍基础
- Agent 只负责填入内容和配置元素
- 管线负责从模板+元素组装最终输出

### 决策5：新平台 = 新管线

**决策**：未来新增平台时，只需编写新管线，元素层不变。

这是三层架构的核心收益：
- 新增小红书图文 → 编写小红书管线（读取文本+图形元素，按小红书约束组装）
- 新增抖音视频 → 编写抖音管线（读取图形+动画+音频元素，按抖音约束组装）
- 新增PDF报告 → 编写PDF管线（读取文本+图形元素，按PDF约束组装）

元素层是稳定的，管线是可扩展的。

### 决策6：品牌主题统一

**决策**：复用视频模板已有的4主题体系，统一到 `brand-themes.js`，所有管线共享。

```javascript
const THEMES = {
  dark: { /* 暗色科技风 */ },
  warm: { /* 暖色人文风 */ },
  minimal: { /* 极简白 */ },
  nature: { /* 自然绿 */ },
};
```

品牌主题是数据元素（`data/brand-spec.json`），所有管线读取同一份主题数据。

## 十二、质量门禁

### 12.1 元素层质检

| 检查项 | 标准 | 严重度 |
|--------|------|--------|
| 元素自包含 | 可独立打开/播放/渲染 | 阻断 |
| manifest注册 | 每个元素在manifest中有记录 | 阻断 |
| SVG有效性 | 浏览器打开无报错，放大200%无模糊 | 阻断 |
| 文本完整性 | Markdown有标题、段落、结构 | 警告 |
| 音频可播放 | MP3文件可正常播放 | 警告 |

### 12.2 管线输出质检

| 检查项 | 标准 | 严重度 |
|--------|------|--------|
| 浏览器打开无报错 | 控制台 0 Error | 阻断 |
| 核心交互可用（HTML管线） | 所有click/hover事件正常 | 阻断 |
| 主题切换正常 | 4主题切换无样式错乱 | 阻断 |
| 移动端不崩 | 375px宽度下布局不错乱 | 阻断 |
| 元素复用验证 | 同一元素在多管线中输出一致 | 警告 |

### 12.3 模板完整性基线

每个HTML模板必须包含：

1. 品牌主题系统（`brand-themes.js`）
2. 响应式 viewport meta
3. 中文字体栈（PingFang SC → Microsoft YaHei → Noto Sans SC）
4. 主题切换控件
5. 标准模块接入点
6. 无障碍基础（lang、alt、aria）

## 十三、与 TeachAny 的差异定位

| 维度 | TeachAny | 本质工坊 |
|------|----------|---------|
| **领域** | K-12 教育 | 认知科学 / 思维蒸馏 |
| **用户** | 教师、家长、学生 | 知识工作者、思考者 |
| **核心循环** | 问题锚点→互动→反馈→迁移 | 三阶推导→坡度理解→输出 |
| **交互范式** | 学习型交互（练/测/评） | 探索型交互（展开/关联/深入） |
| **知识结构** | 课标知识树 | 心智模型网络 |
| **输出架构** | 单一HTML课件 | 元素层→管线层→平台层 |
| **平台适配** | Gallery+CDN | 公众号+视频号+HTML+演示+扩展 |

**核心差异**：TeachAny 的输出是单一格式（HTML课件），本质工坊的输出是多管线多平台。三层架构是本质工坊的独特需求——因为认知内容需要同时交付到公众号（降级）、视频号（录制）、HTML（完整）等不同平台。

## 十四、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 元素层抽象过度 | 简单场景也要走元素→管线流程 | 简单场景可跳过元素层，直接走管线（降级模式） |
| 元素格式不兼容 | SVG在某些管线中转换困难 | SVG优先，Canvas备选；管线负责转换 |
| 管线维护成本 | 管线越多，维护越难 | 管线最小化，共享转换函数；模板优先 |
| AI生成元素质量波动 | SVG结构不规范、交互不可用 | 元素质检 + 模板约束 + 浏览器验证 |
| 现有管线改造成本 | 公众号/视频号管线重构风险 | 渐进改造：新内容走新流程，旧内容保持兼容 |
| 移动端兼容性 | 交互在手机上不可用 | 响应式设计 + 触摸事件 + 降级方案 |

## 十五、总结

本方案的核心架构是三层分离：

```
认知层（不变）：三阶方法论 × 坡度理解 × 类-属性-方法-路由
    ↓ 产出认知内容
元素层（新增核心）：原子资产（SVG/Markdown/JSON/MP3/JS模块）
    ↓ 管线读取
管线层（改造+新增）：公众号管线 / 视频号管线 / HTML交互管线 / 演示管线 / PPT管线 / 未来管线
    ↓ 平台交付
平台层：微信公众号 / 微信视频号 / 浏览器HTML / Reveal.js / PPT(.pptx) / 未来平台
```

**三个核心收益**：

1. **元素复用**：同一张知识图谱，公众号出PNG、视频号出Canvas帧、HTML出交互SVG——改一处，全平台更新
2. **管线扩展**：新增平台 = 新增管线，元素层不变，改造成本最低
3. **能力分层**：公众号是降级交付（静态图），视频号是录制交付（视频帧），HTML交互是完整交付（全部能力）——同一内容，不同深度的消费体验

**变的是交付架构，不变的是认知方法。** 元素层让认知产出从「每条管线独立生成」升级为「统一生成原子资产，多管线按需消费」，从「文本交互」进化为「富媒体交付」。
