# 坡度理解融合方式与跨场景调用规范

> **核心定位**：概念寓言化是坡度理解"阶段2：建立联系"的一种高具象载体，可作为元素层原子资产被多个场景复用。

---

## 一、坡度理解三阶段中的寓言定位

```
阶段1：点出概念    ← 概念名 + 一句话定义（轻量钩子）
阶段2：建立联系    ← 【寓言在此】用叙事把概念嫁接到熟悉情境
阶段3：详细解释    ← 正式定义 + 映射表 + 迁移问题
```

### 寓言 vs 其他"建立联系"载体

| 载体 | 具象度 | 适用受众 | 认知负荷 | 适用场景 |
|------|--------|---------|---------|---------|
| 逻辑类比表格 | 中 | 同行/专业人士 | 低 | 知识探索、项目解析 |
| 生活案例 | 高 | 外行/初学者 | 中 | 内容输出、K12拆解 |
| **寓言故事** | **最高** | **外行/儿童/全受众** | **中** | **全场景通用** |
| 图示/动画 | 高 | 视觉学习者 | 低 | K12拆解、HTML交互 |

**判断规则**：
- 受众是同行 → 用逻辑类比表格即可，不必寓言化
- 受众是外行/儿童 → 优先用寓言
- 受众混合 → 寓言引入 + 类比表格补充

## 二、跨场景调用协议

### 调用接口

所有场景通过统一接口调用概念寓言化：

```
调用方传入：
  concept: 概念名
  audience: 受众类型（child/layman/beginner/expert）
  length: 篇幅偏好（short/medium/long，可选）

概念寓言化返回：
  fable: 寓言正文（含点破句）
  mapping: 寓言→概念映射表
  questions: 2个迁移问题
  element: 元素层封装格式（纯文本 + JSON）
```

### 各场景调用方式

#### 场景A：知识探索

```
A的工作流第1步"是什么"中，阶段2"建立联系"原本用类比表格。
当受众判断为外行时，替换为：
  阶段2：建立联系 = 调用 fable-concept(concept, audience=layman)
  → 寓言替代类比表格，后续阶段3照常详细解释
```

**效果**：降低外行认知门槛，但会增加输出篇幅。同行受众不调用。

#### 场景A2：K12知识拆解

```
A2生成的交互式HTML中，增加"寓言引入"模块：
  在知识图谱节点页面顶部插入：
    <section class="fable-intro">
      {fable-concept 返回的寓言正文}
    </section>
  在交互练习区插入迁移问题
```

**效果**：儿童通过故事先建立直觉，再进入七维框架拆解，理解深度提升。

#### 场景B：蒸馏

```
B蒸馏出的认知操作系统Skill中，表达DNA部分增加"寓言接口"：
  当用户说"用寓言讲讲XX的XX思维"时：
    1. 从蒸馏Skill提取核心心智模型
    2. 调用 fable-concept(model_name, audience=layman)
    3. 用此人的表达DNA改写寓言语气
```

**效果**：蒸馏Skill获得"用故事解释自己思维"的能力，增强传播性。

#### 场景E：内容输出

```
E的"故事风格"文章生成时：
  开头钩子 = fable-concept(concept, audience=layman, length=short)
  → 寓言作为文章开头，后续接观点论证
  → 同一寓言可被公众号/视频号/PPT管线复用
```

**效果**：故事风格文章的开头不再需要人工构思，寓言元素可跨管线复用。

#### 场景E2：连载小说

```
E2的世界观构建时，为每个"概念节点"预生成寓言：
  for each 概念节点 in 世界观.概念图谱:
    fable = fable-concept(概念, audience=layman)
    → 存入章节大纲作为本章情节内核
  → 写章节时，角色冒险情节围绕寓言冲突展开
```

**效果**：连载小说的每章都有概念内核，叙事与知识传递合一。

## 三、组合流价值评估

### 高价值组合流（推荐优先实现）

| 组合流 | 路径 | 价值 | 复用度 |
|--------|------|------|--------|
| **探索→寓言→输出** | A(探索概念) → G(寓言化) → E(公众号/视频) | 一次探索，故事化输出多平台 | ★★★★★ |
| **K12拆解→寓言→HTML** | A2(拆解) → G(寓言引入) → HTML交互 | 儿童先听故事再学知识 | ★★★★☆ |
| **蒸馏→寓言→输出** | B(蒸馏人物) → G(寓言化其思维) → E(输出) | 用故事传播人物思维模型 | ★★★★☆ |
| **连载小说←寓言内核** | G(预生成概念寓言) → E2(章节情节) | 每章一个概念内核 | ★★★☆☆ |

### 中等价值组合流

| 组合流 | 路径 | 价值 | 限制 |
|--------|------|------|------|
| 项目解析→寓言 | D(解析) → G(寓言化架构理念) → E(输出) | 向非技术成员解释架构 | 仅需时调用 |
| 迭代开发→寓言 | C2(迭代) → G(寓言化Ratchet Loop) | 团队对齐方法论 | 一次性 |

### 不推荐组合流

| 组合流 | 原因 |
|--------|------|
| Skill优化→寓言 | F是元操作，寓言化优化原则无实际价值 |
| 项目开发→全程寓言 | C需要精确技术文档，寓言会降低准确性 |

## 四、元素层封装格式

概念寓言化输出必须封装为两种格式，供管线层调用：

### 纯文本格式（公众号/视频/PPT用）

```markdown
## 寓言：{标题}

{寓言正文}

> 这个故事讲的，其实是 **{概念}**。

**概念解释**：{正式定义}

**思考题**：
1. {理解检验问题}
2. {跨域迁移问题}
```

### JSON格式（HTML交互/Notebook/程序调用用）

```json
{
  "element_type": "fable",
  "concept": "{概念名}",
  "audience": "{受众类型}",
  "title": "{寓言标题}",
  "body": "{寓言正文}",
  "reveal": "这个故事讲的，其实是{概念}",
  "definition": "{正式定义}",
  "mapping": [
    {"fable_element": "...", "concept_element": "...", "logic": "..."}
  ],
  "questions": {
    "comprehension": "{理解检验问题}",
    "transfer": "{跨域迁移问题}"
  }
}
```
