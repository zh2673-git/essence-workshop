# 场景F：Skill优化（自主优化循环）

> **设计依据**：借鉴 Darwin Skill 2.0 的 hill-climbing 优化循环 + Karpathy autoresearch 的自主实验思想，适配本质工坊语境简化。
>
> **核心理念**：评估 → 改进 → 实测验证 → 人类确认 → 保留或回滚 → 生成评分卡

---

## 触发条件

- 用户说「优化skill」「skill评分」「达尔文」「darwin」「自动优化」
- 用户说「帮我改改skill」「skill怎么样」「提升skill质量」
- 用户说「skill review」「skill打分」
- 场景B蒸馏完成后，质量门控评分低于A级

---

## Phase 0: 初始化

```
1. 确认优化范围：
   - 本质工坊自身 → 读取 SKILL.md + references/ + workflows/
   - 蒸馏Skill → 扫描 examples/*/SKILL.md
   - 指定Skill → 用户指定路径
2. 创建备份：cp SKILL.md SKILL.md.bak.YYYYMMDD-HHMM
3. 初始化优化记录（如不存在）
4. 读取历史优化记录（如存在）
```

### 失败模式

| 场景 | 处理动作 |
|------|---------|
| SKILL.md不存在 | 告知用户，终止该Skill的优化 |
| 备份失败 | 尝试不同路径备份；仍失败则告知用户，由用户决定是否继续 |
| 文件只读 | 告知用户修改权限，终止 |

## Phase 0.5: 测试Prompt设计

在评估之前，为每个Skill设计测试prompt。没有测试prompt，dim8（实测表现）无法打分。

```
for each skill:
  1. 读取 SKILL.md，理解它做什么
  2. 设计2-3个测试prompt，覆盖：
     - 最典型的使用场景（happy path）
     - 一个稍复杂或有歧义的场景
  3. 保存到 skill目录/test-prompts.json：
     [
       {"id": 1, "prompt": "用户会说的话", "expected": "期望输出的简短描述"},
       {"id": 2, "prompt": "...", "expected": "..."}
     ]
```

展示所有测试prompt给用户，**确认后再进入评估**。

🔵 **AUTO-CHECK · 测试Prompt确认**：输出测试prompt，自动继续（用户可随时打断调整）

---

## Phase 1: 基线评估（Baseline）

```
for each skill in 优化范围:
  # 结构评分
  1. 读取 SKILL.md 全文
  2. 按维度1-7、9-10逐项打分（附简短理由）
  # 效果评分
  3. 对每个测试prompt，执行验证：
     - with_skill: 带着SKILL.md执行测试prompt
     - baseline: 不带skill执行同一prompt
  4. 对比两组输出，打维度8的分
  # 汇总
  5. 计算加权总分
  6. 记录到优化记录
```

**如果无法跑测试**：dim8用干跑验证打分，标注 `dry_run`。不要因为跑不了测试就跳过。

基线评估完成后，展示评分卡：

```
┌──────────────────────┬───────┬──────────────┬──────────────┐
│ Skill                │ Score │ 结构短板      │ 效果短板      │
├──────────────────────┼───────┼──────────────┼──────────────┤
│ nihaixia.skill       │ 78    │ dim3失败模式  │ dim8测试2    │
│ socrates.skill       │ 72    │ dim5具体性    │ baseline持平  │
├──────────────────────┼───────┼──────────────┼──────────────┤
│ 平均                 │ 75    │              │              │
└──────────────────────┴───────┴──────────────┴──────────────┘
```

🔵 **AUTO-CHECK · 基线评估**：输出评分卡，自动进入优化循环（用户可随时打断）

## Phase 2: 优化循环

用户确认后，按基线分数从低到高排序，先优化最弱的。

```
for each skill:
  round = 0
  while round < MAX_ROUNDS (默认3):
    round += 1

    # Step 1: 诊断
    找出得分最低的维度
    # 维度相关簇警告：dim2/3/4是相关簇，修一个时另两个常跟着涨
    # → 不要因为dim3最低就单独修，要看整簇短板再决定是否同步改

    # Step 2: 提出改进方案
    针对最低维度，生成1个具体改进方案：
      - 改什么（具体段落/行）
      - 为什么改（对应rubric哪条）
      - 预期提升多少分

    # Step 3: 执行改进
    编辑 SKILL.md

    # Step 4: 重新评估
    - 结构维度：重新打分
    - 效果维度：重跑测试prompt（关键！不能自己评自己）

    # Step 5: 决策
    if 新总分 > 旧总分:
      status = "keep"，更新旧总分
      # 见好就收：连续2轮Δ<2分→break
      if last_delta < 2.0 and this_delta < 2.0:
        print("触顶信号：连续2轮边际收益<2分，停止优化")
        break
    else:
      status = "revert"
      从备份恢复 SKILL.md
      记录失败尝试
      break  # 该Skill到瓶颈，跳到下一个

    # Step 6: 日志
    追加优化记录

  # === 🔴 CHECKPOINT · 每个 Skill 优化完后强制人审 ===
  展示该Skill的改动摘要：
    - 改前 vs 改后对比
    - 分数变化（哪些维度提升/下降）
    - 测试prompt输出对比
  等用户确认OK再继续下一个Skill。
  如果用户说"不好"，从备份恢复。
```

### Phase 2.5: 探索性重写（按需触发）

当hill-climbing连续2个Skill都在round 1就break（涨不动）时，提议一次「探索性重写」：

```
1. 选一个瓶颈Skill
2. 备份当前版本
3. 从头重写SKILL.md（不是微调，是重新组织结构和表达方式）
4. 重新评估
5. if 重写版 > 备份版: 采用重写版
   else: 从备份恢复
```

🔵 **AUTO-CHECK · 探索性重写**：输出重写方案，自动执行（用户可随时打断）

## 优化策略库

按优先级排序，每轮只做最高优先级的一个：

### P0: 效果问题（实测发现的）
- 测试输出偏离用户意图 → 检查Skill是否有误导性指令
- 带Skill比不带还差 → Skill可能过度约束，考虑精简
- 输出格式不符合预期 → 补充明确的输出模板

### P1: 结构性问题
- Frontmatter缺少触发词 → 补充中英文触发词
- 缺少Phase/Step结构 → 重组为线性流程
- 缺少用户确认检查点 → 在关键决策处插入🔴/🔵

### P2: 具体性问题
- 步骤模糊（"处理图片"）→ 改为具体操作和参数
- 缺少输入/输出规格 → 补充格式、路径、示例
- 缺少异常处理 → 补充"如果X失败，则Y"

### P3: 可读性问题
- 段落过长 → 拆分+用表格
- 重复描述 → 合并去重
- 缺少速查 → 添加TL;DR或决策树

## 失败模式编码

| 场景 | 处理动作 |
|------|---------|
| 备份文件不存在 | 重新创建备份后再操作 |
| SKILL.md为空 | 终止该Skill优化，告知用户 |
| 优化后体积>原150% | 拒绝提交，精简冗余后再评 |
| MAX_ROUNDS触顶 | 展示最弱维度，问用户"继续/探索性重写/收工" |
| 测试prompt执行超时 | 降级为dry_run，标注 |
| 所有维度都已≥8分 | 跳过该Skill，进入下一个 |
